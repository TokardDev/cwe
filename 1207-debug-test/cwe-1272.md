# CWE-1272 - Sensitive Information Uncleared Before Debug/Power State Transition


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2020-12926](https://www.cve.org/CVERecord?id=CVE-2020-12926) | Product software does not set a flag as per TPM specifications, thereby preventing a failed authorization attempt from being recorded after a loss of power. |

## CWE Explanation

This weakness occurs when a hardware product transitions between power or debug states without properly clearing sensitive information that should no longer be accessible in the target state. Modern devices employ numerous power states (e.g., hibernate, sleep, deep sleep) and debug conditions to optimize performance and facilitate diagnosis.

If sensitive data from a more-secure or privileged state is retained in registers, caches, or internal buffers after transitioning to a less-secure or unprivileged state, it may be leaked. For example, a root key read from fuses in a "secure boot" state might remain in a register when the system transitions to a "user debug" state, allowing an attacker to extract it.

The root cause is a failure to identify all security-sensitive resources that must be zeroized or scrubbed as part of the state transition logic within the hardware state machine.

## Impact Assessment

| Aspect | Impact |
|--------|--------|

| **Availability** | Low impact (*No direct impact on availability*) |

## Attack Scenario

An attacker targets a hardware device that uses a specialized power state for secure firmware updates, during which private cryptographic keys are loaded into local memory. In the next step, the device transitions to a low-power "hibernate" state and then resumes to a normal "user" state. Because the hardware power-management controller fails to scrub the local memory during these transitions, the attacker successfully extracts the private keys from the residual memory contents after the system returns to the normal operating state.

## MITRE Reference Data

### Description

The product performs a power or debug state transition, but it does not clear sensitive information that should no longer be accessible due to changes to information access restrictions.

### Extended Description

A device or system frequently employs many power and sleep states during its normal operation (e.g., normal power, additional power, low power, hibernate, deep sleep, etc.). A device also may be operating within a debug condition. State transitions can happen from one power or debug state to another. If there is information available in the previous state which should not be available in the next state and is not properly removed before the transition into the next state, sensitive information may leak from the system.

### Demonstrative Examples

#### Example 1

This example shows how an attacker can take advantage of an incorrect state transition.

Suppose a device is transitioning from state A to state B. During state A, it can read certain private keys from the hidden fuses that are only accessible in state A but not in state B. The device reads the keys, performs operations using those keys, then transitions to state B, where those private keys should no longer be accessible.

During the transition from A to B, the device does not scrub the memory.

After the transition to state B, even though the private keys are no longer accessible directly from the fuses in state B, they can be accessed indirectly by reading the memory that contains the private keys.

For transition from state A to state B, remove information which should not be available once the transition is complete.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation | - | During state transitions, information not needed in the next state should be removed before the transition to the next state. | - |

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design a state-transition scrubbing controller that identifies and zeroizes all security-sensitive registers, caches, and buffers before completing any power or debug state transition | Critical |
| Implementation | Maintain a hardware-enforced checklist of all memory regions and registers containing sensitive data, with mandatory scrub verification before state transition completes | Critical |
| Testing | Validate memory scrubbing across all supported power state transitions (sleep, hibernate, deep sleep, debug) in post-silicon testing | High |
| Architecture and Design | Implement read-back verification after scrubbing to confirm that all sensitive locations have been properly cleared before the transition is finalized | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
