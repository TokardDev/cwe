# CWE-1323 - Improper Management of Sensitive Trace Data

## CWE Explanation

This weakness occurs when a System-on-Chip (SoC) fails to properly protect sensitive trace data collected from internal IP blocks. Tracing infrastructure is used to monitor internal signals in real-time for debugging and performance analysis. These traces often include highly sensitive information, such as CPU instruction streams, memory operands, cryptographic keys, and intermediate values from security coprocessors.

If trace data is stored in unprotected memory locations or transported to untrusted debug agents without proper access control, it can be intercepted by an attacker. For instance, if traces from a secure enclave are aggregated with non-secure traces and sent to a common debug port, an unprivileged debugger could extract the enclave's secrets.

The root cause is a failure to manage the "security lifecycle" of trace data, specifically failing to tag traces with appropriate privilege levels at the source and verify those tags at the sink (the debugger interface).

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | High - Attacker can read sensitive system behavior, cryptographic keys, and user data captured in traces. |
| **Integrity** | Low impact (*Mainly focused on information disclosure*) |
| **Availability** | Low impact (*No direct impact on availability*) |

## Attack Scenario

An attacker connects an unprivileged software debugger to an SoC's trace buffer. In the next step, the attacker discovers that the hardware trace aggregator incorrectly mixes trace data from the general-purpose CPU with data from a secure cryptographic coprocessor. Because the aggregator fails to enforce any privilege-based filtering or tagging, the attacker successfully extracts real-time traces of cryptographic key loads and intermediate cipher states, compromising the device's root of trust.

## MITRE Reference Data

### Description

Trace data collected from several sources on the System-on-Chip (SoC) is stored in unprotected locations or transported to untrusted agents.

### Extended Description

To facilitate verification of complex System-on-Chip (SoC) designs, SoC integrators add specific IP blocks that trace the SoC's internal signals in real-time. This infrastructure enables observability of the SoC's internal behavior, validation of its functional design, and detection of hardware and software bugs. Such tracing IP blocks collect traces from several sources on the SoC including the CPU, crypto coprocessors, and on-chip fabrics. Traces collected from these sources are then aggregated inside trace IP block and forwarded to trace sinks, such as debug-trace ports that facilitate debugging by external hardware and software debuggers.

Since these traces are collected from several security-sensitive sources, they must be protected against untrusted debuggers. If they are stored in unprotected memory, an untrusted software debugger can access these traces and extract secret information. Additionally, if security-sensitive traces are not tagged as secure, an untrusted hardware debugger might access them to extract confidential information.

### Demonstrative Examples

#### Example 1

In a SoC, traces generated from sources include security-sensitive IP blocks such as CPU (with tracing information such as instructions executed and memory operands), on-chip fabric (e.g., memory-transfer signals, transaction type and destination, and on-chip-firewall-error signals), power-management IP blocks (e.g., clock- and power-gating signals), and cryptographic coprocessors (e.g., cryptographic keys and intermediate values of crypto operations), among other non-security-sensitive IP blocks including timers and other functional blocks. The collected traces are then forwarded to the debug and trace interface used by the external hardware debugger.

The traces do not have any privilege level attached to them. All collected traces can be viewed by any debugger (i.e., SoC designer, OEM debugger, or end user).

Some of the traces are SoC-design-house secrets, while some are OEM secrets. Few are end-user secrets and the rest are not security-sensitive. Tag all traces with the appropriate, privilege level at the source. The bits indicating the privilege level must be immutable in their transit from trace source to the final, trace sink. Debugger privilege level must be checked before providing access to traces.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Implementation | - | Tag traces to indicate owner and debugging privilege level (designer, OEM, or end user) needed to access that trace. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Tag all trace data at the source with immutable privilege-level indicators (e.g., secure/non-secure, designer/OEM/end-user) that cannot be modified in transit | Critical |
| Architecture and Design | Implement hardware-enforced filtering at the trace aggregator that blocks security-sensitive traces from reaching unauthorized debug sinks | Critical |
| Implementation | Store trace data in access-controlled memory regions with permissions aligned to the trace privilege level | High |
| Testing | Validate trace data isolation by attempting to access secure traces from unprivileged debug interfaces in post-silicon testing | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
