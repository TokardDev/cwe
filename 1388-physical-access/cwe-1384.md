# CWE-1384 - Improper Handling of Physical or Environmental Conditions


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2019-17391](https://www.cve.org/CVERecord?id=CVE-2019-17391) | Lack of anti-glitch protections allows an attacker to launch a physical attack to bypass the secure boot and read protected eFuses. |

## CWE Explanation

This weakness describes the failure of a hardware product to maintain security properties when subjected to unexpected physical or environmental conditions. Hardware is typically validated for specific operating ranges (voltage, temperature, clock frequency). However, when these boundaries are exceeded—either due to natural phenomena or active manipulation by an adversary—the device may behave unpredictably.

Common vectors include introducing voltage drops, clock glitches, extreme temperatures, electromagnetic interference, or focused radiation (lasers, ion beams). If the hardware logic does not include sensors or defensive designs to detect these anomalies and shut down safely, the environmental stress can cause bit flips, instruction skips, or undefined behavior. Attackers exploit this to bypass authentication, disable security features, or force the device into a vulnerable debug state.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can bypass access controls or readout protections to access secrets |
| **Integrity** | The attacker can force the execution of unauthorized code or modify critical data |

## Attack Scenarios

### Scenario 1
An attacker targets the secure boot sequence of an embedded device. By analyzing the power trace, they locate the exact instruction where the bootloader verifies the digital signature of the firmware. Just as the processor attempts to execute the conditional branch associated with this check, the attacker induces a rapid voltage glitch on the power rail. The hardware, lacking anti-glitch sensors, misinterprets the logic level, causing the processor to "fall through" the security check and execute the unauthorized code despite the invalid signature.

### Scenario 2
An attacker exploits the sensitivity of a medical implant to environmental radiation (e.g., cosmic rays at high altitude). The device utilizes non-volatile memory to store its operating mode but lacks error correction for Single Event Upsets (SEUs). A particle strike flips a bit in the status register, triggering the device to enter a "fail-safe" or "safe mode." While intended to protect the patient, this mode inadvertently unlocks a diagnostic interface. The attacker then connects to this interface to extract the device's hard-coded encryption keys.

## MITRE Reference Data

### Description

The product does not properly handle unexpected physical or environmental conditions that occur naturally or are artificially induced.

### Extended Description

Hardware products are typically only guaranteed to behave correctly within certain physical limits or environmental conditions. Such products cannot necessarily control the physical or external conditions to which they are subjected. However, the inability to handle such conditions can undermine a product's security. For example, an unexpected physical or environmental condition may cause the flipping of a bit that is used for an authentication decision. This unexpected condition could occur naturally or be induced artificially by an adversary.

Physical or environmental conditions of concern are:
* Atmospheric characteristics: extreme temperature ranges, etc.
* Interference: electromagnetic interference (EMI), radio frequency interference (RFI), etc.
* Assorted light sources: white light, ultra-violet light (UV), lasers, infrared (IR), etc.
* Power variances: under-voltages, over-voltages, under-current, over-current, etc.
* Clock variances: glitching, overclocking, clock stretching, etc.
* Component aging and degradation
* Materials manipulation: focused ion beams (FIB), etc.
* Exposure to radiation: x-rays, cosmic radiation, etc.

#### Example 1
Below is a representative snippet of C code that is part of the secure-boot flow. A signature of the runtime-firmware image is calculated and compared against a golden value. If the signatures match, the bootloader loads runtime firmware. If there is no match, an error halt occurs. If the underlying hardware executing this code does not contain any circuitry or sensors to detect voltage or clock glitches, an attacker might launch a fault-injection attack right when the signature check is happening (at the location marked with the comment), causing a bypass of the signature-checking process.


After bypassing secure boot, an attacker can gain access to system assets to which the attacker should not have access.

```other

If the underlying hardware detects a voltage or clock glitch, the information can be used to prevent the glitch from being successful.
```

#### Example 2
In 2016, a security researcher, who was also a patient using a pacemaker, was on an airplane when a bit flip occurred in the pacemaker, likely due to the higher prevalence of cosmic radiation at such heights. The pacemaker was designed to account for bit flips and went into a default safe mode, which still forced the patient to go to a hospital to get it reset. The bit flip also inadvertently enabled the researcher to access the crash file, perform reverse engineering, and detect a hard-coded key. [REF-1101]

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Requirements | - | In requirements, be specific about expectations for how the product will perform when it exceeds physical and environmental boundary conditions, e.g., by shutting down. | - |
| Architecture and Design; Implementation | - | Where possible, include independent components that can detect excess environmental conditions and have the capability to shut down the product. | - |
| Architecture and Design; Implementation | - | Where possible, use shielding or other materials that can increase the adversary's workload and reduce the likelihood of being able to successfully trigger a security-related failure. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Integrate multi-parameter environmental monitoring (voltage, clock frequency, temperature, EM field) with hardware-enforced safe shutdown or security lockdown when any parameter exceeds validated operating ranges. | Critical |
| Architecture and Design | Design all security-critical decision paths with redundancy (dual-check logic, TMR for critical registers) so that a single environmentally-induced fault cannot bypass authentication, signature verification, or access control. | Critical |
| Implementation | Implement automatic key and secret zeroization triggered by tamper detection sensors, ensuring that environmentally-induced faults cannot be exploited to extract sensitive data. | High |
| Requirements | Define explicit security behavior specifications for out-of-range environmental conditions, including graceful degradation modes that do not expose debug interfaces or weaken access controls. | High |
| Testing | Perform comprehensive environmental stress testing (voltage glitching, clock manipulation, temperature extremes, EMFI) against all security-critical functions to validate that protective mechanisms respond correctly. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
