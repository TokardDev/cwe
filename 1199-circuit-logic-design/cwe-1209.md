# CWE-1209 - Failure to Disable Reserved Bits

## CWE Explanation

Reserved bits in hardware registers or address spaces are typically intended for future use or expansion and should not have any functional effect in the current design. This weakness occurs when these bits are connected to active logic—often for debugging, testing, or enabling future features—but are not disabled or disconnected in the production release.

Designers sometimes utilize reserved bits to hide experimental features or diagnostic controls, intending to speed up time-to-market or facilitate post-silicon validation. If this logic remains active in the final product, it creates a security vulnerability. An attacker who discovers these hidden functions can write to the "reserved" bits to trigger undocumented behavior, bypass security controls, or disrupt system operation.

The root cause is a failure to properly disable or remove the logic associated with these bits before the design is finalized for production. This leaves a "backdoor" or unintended interface that contradicts the documentation stating the bits are reserved and non-functional.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can extract confidential data if the reserved bits enable debug readout or access to restricted memory. |
| **Integrity** | The attacker can modify system state or configuration by activating undocumented hardware features. |
| **Availability** | The attacker can cause system instability, resets, or shutdowns by triggering untested or dangerous hardware states. |

## Attack Scenario

An attacker identifies a memory-mapped register described in the datasheets as having "reserved" bits. By writing various patterns to these bits, the attacker discovers that writing a specific value to the reserved field triggers a change in the hardware's behavior, such as asserting a General Purpose I/O (GPIO) pin or enabling a debug mode. This occurs because the hardware designers left the logic associated with these bits active in the production silicon. The attacker exploits this undocumented interface to manipulate the hardware state, potentially bypassing security checks or causing a denial of service, depending on the functionality linked to the reserved bits.

## MITRE Reference Data

### Description

The reserved bits in a hardware design are not disabled prior to production. Typically, reserved bits are used for future capabilities and should not support any functional logic in the design.   However, designers might covertly use these bits to debug or further develop new capabilities in production hardware. Adversaries with access to these bits will write to them in hopes of compromising hardware state.

### Extended Description

Reserved bits are labeled as such so they can be allocated for a later purpose. They are not to do anything in the current design.  However, designers might want to use these bits to debug or control/configure a future capability to help minimize time to market (TTM). If the logic being controlled by these bits is still enabled in production, an adversary could use the logic to induce unwanted/unsupported behavior in the hardware.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation | - | Include a feature to disable reserved bits. | - |
| Integration | - | Any writes to these reserve bits are blocked (e.g., ignored, access-protected, etc.), or an exception can be asserted. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Ensure all reserved bit fields in register maps default to zero and are hardwired to have no functional effect on any logic path. | Critical |
| Implementation | Add explicit `default` cases in RTL case statements and tie reserved address decode outputs to no-ops or access faults. | Critical |
| Verification | Create directed test cases that exhaustively write all possible values to reserved fields and verify no observable state change occurs. | High |
| Integration | Implement bus-level firewalls or register access filters that silently drop or fault on writes to reserved bit positions. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
