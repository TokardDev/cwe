# CWE-1231 - Improper Prevention of Lock Bit Modification


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2017-6283](https://www.cve.org/CVERecord?id=CVE-2017-6283) | chip reset clears critical read/write lock permissions for RSA function |

## CWE Explanation

This weakness occurs when a hardware design employs a "lock bit" mechanism to protect critical registers or memory regions but fails to prevent the lock bit itself from being cleared or modified by unauthorized means. Lock bits are a standard security primitive: once set by trusted firmware (e.g., during boot), they should render the protected resources immutable until the next full system reset.

However, design flaws can allow software or hardware events to inadvertently reset or clear this lock bit. A common issue is connecting the lock bit's reset logic to a non-global reset signal, such as a peripheral-level or soft reset. If an attacker can trigger this specific reset line—perhaps by crashing a driver or requesting a debug reset—the lock bit clears while the system remains powered on. This effectively unlocks the protected resources, allowing the attacker to modify critical security settings, disable protections, or alter immutable data.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | The primary impact is the bypass of access controls. Protected resources become writable again after the lock is illicitly cleared. |
| **Integrity** | Critical system configurations or code regions assumed to be static can be modified, compromising system integrity. |
| **Availability** | An attacker could unlock and then corrupt configuration registers, leading to system crashes or unpredictable behavior. |

## Attack Scenario

A System-on-Chip (SoC) uses a central "Registry Lock" module to protect security-critical configuration registers. The design intends for these locks to persist until a full "Power-On Reset" (POR). However, the hardware logic for the registry lock module mistakenly includes a "Debug Reset" or a specific "Peripheral Reset" signal in its reset condition. An attacker, running unprivileged code, triggers a crash or issues a command that asserts this specific reset signal. The system does not fully reboot, but the Registry Lock module interprets the signal as a reset command and clears all lock bits. The attacker then writes to the previously locked security registers, disabling secure boot enforcement or granting themselves expanded privileges.

## MITRE Reference Data

### Description

The product uses a trusted lock bit for restricting access to registers, address regions, or other resources, but the product does not prevent the value of the lock bit from being modified after it has been set.

### Extended Description

In integrated circuits and hardware intellectual property (IP) cores, device configuration controls are commonly programmed after a device power reset by a trusted firmware or software module (e.g., BIOS/bootloader) and then locked from any further modification.

This behavior is commonly implemented using a trusted lock bit. When set, the lock bit disables writes to a protected set of registers or address regions. Design or coding errors in the implementation of the lock bit protection feature may allow the lock bit to be modified or cleared by software after it has been set. Attackers might be able to unlock the system and features that the bit is intended to protect.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation; Testing | - | Security lock bit protections must be reviewed for design inconsistency and common weaknesses. Security lock programming flow and lock properties must be tested in pre-silicon and post-silicon testing. | High |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Lock bit registers must only be resettable by the global Power-On Reset (POR) signal. Exclude all peripheral, soft, and debug reset signals from the lock bit reset condition. | Critical |
| Implementation | Implement lock bits as write-1-to-set, no-clear registers: once the bit is set to 1, no software write of 0 should clear it; only POR resets the bit. | Critical |
| Verification | Use formal verification to prove that no reachable hardware state (including debug, scan, or soft reset) can clear or modify a set lock bit outside of a full power-on reset. | High |
| Testing | In post-silicon testing, attempt to clear lock bits through all available reset vectors (JTAG reset, watchdog reset, peripheral reset) and verify they remain set. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
