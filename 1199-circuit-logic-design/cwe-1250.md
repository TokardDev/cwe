# CWE-1250 - Improper Preservation of Consistency Between Independent Representations of Shared State

## CWE Explanation

This weakness occurs in systems where state information is replicated across multiple distributed components—such as caches, shadow Memory Management Units (MMUs), or coherency directories—but the mechanisms to maintain consistency between these copies are insufficient. In complex hardware designs like multi-core System-on-Chips (SoCs), performance optimization often requires local copies of shared configuration data to avoid bottlenecks at a central controller.

However, if the propagation of updates to these local copies is not atomic, prioritized, or guaranteed to complete within a specific window, "time-of-check to time-of-use" (TOCTOU) race conditions can emerge. A central authority might revoke a permission (e.g., disable access to a memory region), but due to network congestion or lack of prioritization, the "revoke" message arrives late to a distant component. During this inconsistency window, the distant component continues to operate under the old, permissive policy, potentially allowing an attacker to access restricted resources or execute unauthorized operations.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | Components operating on stale state information may grant access to resources that have been revoked by the central policy. |
| **Integrity** | Inconsistent views of memory mapping or cache coherency can lead to data corruption or the use of stale data values. |
| **Availability** | Inconsistencies can lead to system-level confusion, deadlocks, or crashes if components make conflicting decisions based on differing views of the state. |

## Attack Scenario

A multi-core processor uses a central Memory Management Unit (MMU) and several "shadow" MMUs distributed near each core to speed up memory access checks. When the operating system decides to revoke access to a sensitive memory page (e.g., containing cryptographic keys), the central MMU updates its table and broadcasts an "invalidate" message to all shadow MMUs.

However, the on-chip interconnect does not prioritize these invalidation messages over general data traffic. An attacker, anticipating the revocation, floods the interconnect with high-bandwidth traffic from another core. This congestion delays the delivery of the invalidation message to the shadow MMU protecting the victim core. During this delay, the attacker's malicious process on the victim core continues to access the sensitive page because the local shadow MMU still believes the access is valid. The attacker successfully extracts the keys before the system eventually converges to a consistent state.

## MITRE Reference Data

### Description

The product has or supports multiple distributed components or sub-systems that are each required to keep their own local copy of shared data - such as state or cache - but the product does not ensure that all local copies remain consistent with each other.

### Extended Description

In highly distributed environments, or on systems with distinct physical components that operate independently, there is often a need for each component to store and update its own local copy of key data such as state or cache, so that all components have the same "view" of the overall system and operate in a coordinated fashion.  For example, users of a social media service or a massively multiplayer online game might be using their own personal computers while also interacting with different physical hosts in a globally distributed service, but all participants must be able to have the same "view" of the world.  Alternately, a processor's Memory Management Unit (MMU) might have "shadow" MMUs to distribute its workload, and all shadow MMUs are expected to have the same accessible ranges of memory.

In such environments, it becomes critical for the product to ensure that this "shared state" is consistently modified across all distributed systems. If state is not consistently maintained across all systems, then critical transactions might take place out of order, or some users might not get the same data as other users.  When this inconsistency affects correctness of operations, it can introduce vulnerabilities in mechanisms that depend on consistent state.

### Demonstrative Examples

<p class="subheading">Example 1</p>
<br/>

<p>Suppose a processor's Memory Management Unit (MMU) has 5 other shadow MMUs to distribute its workload for its various cores. Each MMU has the start address and end address of "accessible" memory. Any time this accessible range changes (as per the processor's boot status), the main MMU sends an update message to all the shadow MMUs.</p>






<p>Suppose the interconnect fabric does not prioritize such "update" packets over other general traffic packets. This introduces a race condition. If an attacker can flood the target with enough messages so that some of those attack packets reach the target before the new access ranges gets updated, then the attacker can leverage this scenario.</p>
