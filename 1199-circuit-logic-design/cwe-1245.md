# CWE-1245 - Improper Finite State Machines (FSMs) in Hardware Logic


## Related CVEs (from NVD)

| CVE | Description |
|-----|-------------|
| [CVE-2024-58311](https://www.cve.org/CVERecord?id=CVE-2024-58311) | Dormakaba Saflok System 6000 predictable key generation from improper FSM |

## CWE Explanation

Finite State Machines (FSMs) are fundamental building blocks in hardware logic, controlling the flow of data and operations based on a defined set of states and transitions. This weakness arises when an FSM implementation fails to account for all possible input combinations or state transitions, particularly "undefined" or "illegal" states.

In Verilog or VHDL, FSMs are often implemented using `case` statements. If a developer forgets to include a `default` clause or explicitly handle every possible bit combination of the state variable or input signals, the hardware behavior becomes undefined when an unexpected input occurs. This can cause the FSM to enter a "stuck" state, transition to an insecure state, or cause the entire system to hang or reset. Attackers can exploit this by injecting crafted inputs (e.g., through an external interface or by manipulating memory mapped to the FSM inputs) to trigger these undefined transitions, leading to Denial of Service (DoS) or potentially bypassing security checks that rely on the FSM being in a specific, known state.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Availability** | The most common impact is a system crash, hang, or deadlock. The FSM enters a state from which it cannot exit, requiring a hard reset. |
| **Access Control** | In some cases, an undefined transition might bypass a "Locked" or "Checking" state, allowing the FSM to jump directly to an "Authorized" or "Grant" state. |
| **Integrity** | If the FSM controls data flow, entering an unknown state could corrupt data or allow writes to happen at the wrong time. |

## Attack Scenario

A hardware security module uses a 3-bit FSM to manage the authentication process. The valid states are defined as 000 (Idle), 001 (Receive Key), 010 (Verify), and 011 (Grant Access). The remaining states (100-111) are unused and not explicitly handled in the logic. An attacker finds a way to glitch the clock or supply an unexpected input sequence that causes the state register to flip to `100`. Because the logic for state `100` is undefined (or defaults to "preserve previous values" on some signals), the `Grant Access` signal might remain asserted from a previous operation, or the system might enter a deadlock where it refuses all further valid inputs. The attacker thus achieves a Denial of Service or potentially maintains access when they should have been de-authenticated.

## MITRE Reference Data

### Description

Faulty finite state machines (FSMs) in the hardware logic allow an attacker to put the system in an undefined state, to cause a denial of service (DoS) or gain privileges on the victim's system.

### Extended Description

The functionality and security of the system heavily depend on the implementation of FSMs. FSMs can be used to indicate the current security state of the system. Lots of secure data operations and data transfers rely on the state reported by the FSM.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation | - | Define all possible states and handle all unused states through default statements. Ensure that system defaults to a secure state. | High |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Implementation | Add explicit `default` clauses in all FSM `case` statements that transition to a known safe/idle state and assert an error flag. | Critical |
| Architecture and Design | Use one-hot or Gray encoding for security-critical FSMs to minimize the risk of illegal state transitions from single bit flips. | High |
| Verification | Run formal verification with reachability analysis to prove that no undefined or illegal FSM state is reachable from any valid state under any input combination. | High |
| Testing | Inject fault conditions in pre-silicon simulation (e.g., forced state register corruption) and verify the FSM recovers to a safe state within a bounded number of clock cycles. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
