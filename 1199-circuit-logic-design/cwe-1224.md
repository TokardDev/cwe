# CWE-1224 - Improper Restriction of Write-Once Bit Fields

## CWE Explanation

This weakness involves the incorrect implementation of "write-once" or "sticky" bits in hardware registers. These bits are designed to allow a single write operation after system reset—typically used by trusted boot firmware to set immutable security configurations—and then lock until the next reset. The vulnerability occurs when the hardware logic fails to strictly enforce this "one-time write" rule under all conditions.

A common implementation error is creating a "write-1-once" field instead of a true "write-once" field. In this flawed design, the lock mechanism is only activated if the specific bit is written to '1'. If software writes a '0' to the bit, the lock does not engage, allowing subsequent writes. This effectively defeats the purpose of the mechanism if the '0' state is meant to be a permanent security choice, or if the attacker can repeatedly write '0's to keep the register mutable until they are ready to exploit it.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | Attackers can modify access control lists or permission bits that were intended to be immutable, granting unauthorized access. |
| **Integrity** | The integrity of the system configuration is violated because settings that should be fixed at boot can be changed at runtime. |
| **Availability** | Critical system settings could be altered during operation, leading to instability or denial of service. |

## Attack Scenario

A hardware register controls a critical security feature, such as a "Disable JTAG" bit. The designers intended for this bit to be write-once: the bootloader decides whether to disable JTAG and writes the bit, locking it. However, the hardware logic is implemented such that the "lock" signal is only asserted if the written data bit is '1'. An attacker who wants to keep JTAG enabled (logic '0') can write a '0' to the register. Because the data is not '1', the lock is not triggered. The register remains writable. Later, the attacker can continue to modify this or other fields in the register, or simply leave it in the unlocked '0' state, bypassing the intended security policy that assumed the register would be immutable after the first write transaction, regardless of the data value.

## MITRE Reference Data

### Description

The hardware design control register "sticky bits" or write-once bit fields are improperly implemented, such that they can be reprogrammed by software.

### Extended Description

Integrated circuits and hardware IP software programmable controls and settings are commonly stored in register circuits. These register contents have to be initialized at hardware reset to define default values that are hard coded in the hardware description language (HDL) code of the hardware unit. A common security protection method used to protect register settings from modification by software is to make the settings write-once or "sticky." This allows writing to such registers only once, whereupon they become read-only. This is useful to allow initial boot software to configure systems settings to secure values while blocking runtime software from modifying such hardware settings.

Failure to implement write-once restrictions in hardware design can expose such registers to being re-programmed by software and written multiple times. For example, write-once fields could be implemented to only be write-protected if they have been set to value "1", wherein they would work as "write-1-once" and not "write-once".

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | During hardware design, all register write-once or sticky fields must be evaluated for proper configuration. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Implementation | Implement true write-once semantics: set the lock status unconditionally on the first write transaction (i.e., `Write_once_status <= 1'b1`), regardless of the data value written. | Critical |
| Verification | Create formal verification assertions proving that after any single write to a write-once field, all subsequent writes are ignored regardless of data content. | Critical |
| Architecture and Design | Document and review the distinction between "write-once" and "write-1-once" for every sticky bit in the register specification, ensuring the correct semantic is implemented in RTL. | High |
| Testing | Include post-silicon test cases that attempt multiple writes with different data patterns (including all-zeros) to every write-once register and verify immutability after the first write. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
