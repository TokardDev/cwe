# CWE-1223 - Race Condition for Write-Once Attributes

## CWE Explanation

This weakness occurs when a "write-once" register or hardware attribute—intended to be immutable after its initial configuration—is subject to a race condition between trusted and untrusted agents. In many hardware designs, security-critical settings (like locking memory regions or disabling debug interfaces) are implemented as "write-once" fields: the first write sets the value and locks the register until the next system reset.

The vulnerability arises when an untrusted software component or hardware agent runs in parallel with the trusted boot process. If the untrusted agent can issue a write command to the register before the trusted software performs its initial configuration, the untrusted agent "wins" the race. It successfully sets the malicious value and locks the register. The trusted software's subsequent write attempt is ignored (since the register is now locked), potentially leaving the system in an insecure state without the trusted software realizing it.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | An attacker can bypass security protections by locking configuration registers with insecure values before the trusted system can secure them. |
| **Integrity** | The integrity of the system configuration is compromised, as the hardware state does not reflect the intended security policy. |
| **Availability** | An attacker could lock a register with a value that disables critical functionality or causes the system to malfunction. |

## Attack Scenario

A System-on-Chip (SoC) has a "write-once" configuration register that controls access to a secure memory region. The design intends for the trusted bootloader to write to this register immediately after reset to lock down the memory. However, the system also has a secondary processor or a DMA controller that is active during the boot process and controlled by less privileged firmware. An attacker, having compromised the secondary firmware, programs it to continuously attempt to write a "unlock" value to the configuration register as soon as the system powers on. If the attacker's write transaction reaches the register before the bootloader's write transaction, the register locks in the "unlocked" state. When the bootloader subsequently attempts to write "lock," the operation fails silently or generates an ignored error, leaving the secure memory exposed.

## MITRE Reference Data

### Description

A write-once register in hardware design is programmable by an untrusted software component earlier than the trusted software component, resulting in a race condition issue.

### Extended Description

Integrated circuits and hardware IP software programmable controls and settings are commonly stored in register circuits. These register contents have to be initialized at hardware reset to defined default values that are hard coded in the hardware description language (HDL) code of the hardware unit. A common security protection method used to protect register settings from modification by software is to make them write-once. This means the hardware implementation only allows writing to such registers once, and they become read-only after having been written once by software. This is useful to allow initial boot software to configure systems settings to secure values while blocking runtime software from modifying such hardware settings.

Implementation issues in hardware design of such controls can expose such registers to a race condition security flaw. For example, consider a hardware design that has two different software/firmware modules executing in parallel. One module is trusted (module A) and another is untrusted (module B). In this design it could be possible for Module B to send write cycles to the write-once register before Module A. Since the field is write-once the programmed value from Module A will be ignored and the pre-empted value programmed by Module B will be used by hardware.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | During hardware design, all register write-once or sticky fields must be evaluated for proper configuration. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Ensure write-once registers are only writable by a single, well-defined trusted agent (e.g., the secure boot ROM) and that no other bus master can issue writes before the trusted agent completes initialization. | Critical |
| Architecture and Design | Use hardware-enforced boot sequencing (e.g., holding secondary cores and DMA engines in reset) until all write-once security registers have been configured and locked by the primary trusted boot agent. | Critical |
| Verification | Develop pre-silicon formal verification properties that prove no untrusted agent can write to a write-once register before the trusted agent does in any reachable state. | High |
| Implementation | Add hardware readback and status reporting so that trusted firmware can verify the actual written value of write-once registers matches the intended value before proceeding with boot. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
