# CWE-1221 - Incorrect Register Defaults or Module Parameters

## CWE Explanation

This weakness occurs when hardware description language (HDL) code defines insecure default values for registers or configuration parameters. In hardware designs, registers often hold security-critical settings, such as access control policies, lock bits, or feature enable/disable flags. These registers must be initialized to a secure state upon system reset. Similarly, parameterized modules (IP blocks) often rely on instantiation parameters to configure memory sizes, base addresses, or security modes.

If these defaults or parameters are set incorrectly in the source code, the hardware will power on or reset into an insecure state. For example, a "secure mode" bit might default to disabled, or a memory range might default to being readable by all. Since these values are hardcoded into the silicon logic (RTL), they cannot be easily patched with software updates. This forces the boot software to race against potential attackers to lock down the system immediately after reset, creating a window of vulnerability or leaving the system permanently exposed if the software fails to reconfigure the hardware correctly.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Confidential data may be exposed if default settings allow unrestricted read access to protected memory regions. |
| **Integrity** | Integrity controls may be bypassed if write-protection defaults are disabled, allowing unauthorized modification of code or data. |
| **Availability** | System availability can be compromised if incorrect parameters cause misconfiguration of peripherals or memory maps, leading to crashes or lockups. |
| **Access Control** | Access control mechanisms may be ineffective at boot time, allowing unauthorized peripherals or software to access restricted resources. |

## Attack Scenario

An attacker targets a System-on-Chip (SoC) where the hardware designers inadvertently set the default value of a critical access control register to "unlock" or "allow all." When the device powers up or resets, the hardware enforces this insecure policy. Even if the trusted boot firmware attempts to configure the register to a secure state later in the boot process, there exists a window of time immediately after reset where the attacker—potentially using a connected peripheral or malicious code running on a different core—can access sensitive assets, dump secrets, or inject code before the protection is enabled. In some cases, if the register is "write-once" and defaults to the wrong state, the secure firmware may be unable to enable the protection at all.

## MITRE Reference Data

### Description

Hardware description language code incorrectly defines register defaults or hardware Intellectual Property (IP) parameters to insecure values.

### Extended Description

Integrated circuits and hardware IP software programmable controls and settings are commonly stored in register circuits. These register contents have to be initialized at hardware reset to defined default values that are hard coded in the hardware description language (HDL) code of the hardware unit. Hardware descriptive languages also support definition of parameter variables, which can be defined in code during instantiation of the hardware IP module. Such parameters are generally used to configure a specific instance of a hardware IP in the design.

The system security settings of a hardware design can be affected by incorrectly defined default values or IP parameters. The hardware IP would be in an insecure state at power reset, and this can be exposed or exploited by untrusted software running on the system. Both register defaults and parameters are hardcoded values, which cannot be changed using software or firmware patches but must be changed in hardware silicon. Thus, such security issues are considerably more difficult to address later in the lifecycle. Hardware designs can have a large number of such parameters and register defaults settings, and it is important to have design tool support to check these settings in an automated way and be able to identify which settings are security sensitive.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | During hardware design, all the system parameters and register defaults must be reviewed to identify security sensitive settings. | - |
| Implementation | - | The default values of these security sensitive settings need to be defined as part of the design review phase. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Establish a "secure by default" policy: all access control registers must reset to the most restrictive state (deny-all), requiring trusted firmware to explicitly open access. | Critical |
| Implementation | Use automated RTL linting rules to flag any security-sensitive register whose reset value does not match the approved secure default from the security specification. | Critical |
| Verification | Create a register default audit checklist and run automated post-synthesis checks comparing actual reset values against the security specification for every tagged register. | High |
| Integration | Implement boot-time attestation that reads back all security-critical register values after firmware configuration and compares them to expected values before enabling untrusted code execution. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
