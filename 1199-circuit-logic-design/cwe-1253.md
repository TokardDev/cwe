# CWE-1253 - Incorrect Selection of Fuse Values

## CWE Explanation

This weakness occurs when a hardware security mechanism relies on the default, unblown state of a fuse (typically logic '0') to represent a "secure" configuration, while the blown state (typically logic '1') represents an "insecure" or "debug" configuration. One-time programmable (OTP) fuses are standard components in modern chips, used to store permanent configuration data. Crucially, the transition from unblown to blown is irreversible and can often be triggered by an attacker with physical access or, in some cases, software access to the fuse controller.

If a design uses "logic 0" (unblown) to mean "Secure Boot Enabled" and "logic 1" (blown) to mean "Secure Boot Disabled," an attacker can simply blow the fuse to disable the security feature permanently. Because blowing a fuse is an additive process (energy is added to break a connection or change a state), it is much easier for an attacker to force a fuse to the blown state than to revert it. A secure design should interpret the blown state as the restrictive/secure state, or use complex encoding schemes (like Hamming distance checks) that prevent simple transitions from valid-secure to valid-insecure states.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | An attacker can permanently disable security mechanisms (like Secure Boot or JTAG locking) by blowing the corresponding control fuses. |
| **Authorization** | Privileged features or debug modes meant to be disabled in production can be reactivated if their control logic relies on unblown fuses. |
| **Integrity** | The integrity of the platform's security state is compromised, as the hardware no longer enforces the intended protections. |

## Attack Scenario

A trusted device utilizes Secure Boot to verify the signature of the bootloader before execution. The hardware logic determines whether to enforce Secure Boot based on the state of a specific fuse named `SECURE_BOOT_ENABLE`. The designers, assuming that the factory defaults (all 0s) should represent the "standard" secure state, implement the logic such that `0` means "Enable Secure Boot" and `1` means "Disable Secure Boot (Development Mode)."

An attacker gains physical access to the device or exploits a vulnerability in the fuse controller driver. They deliberately trigger the programming sequence to blow the `SECURE_BOOT_ENABLE` fuse to `1`. Upon the next reboot, the hardware sees the logic `1`, assumes the device is in Development Mode, and bypasses the signature verification. The attacker can now flash and run a malicious, unsigned bootloader, permanently compromising the device's chain of trust.

## MITRE Reference Data

### Description

The logic level used to set a system to a secure state relies on a fuse being unblown.

### Extended Description

Fuses are often used to store secret data, including security configuration data. When not blown, a fuse is considered to store a logic 0, and, when blown, it indicates a logic 1. Fuses are generally considered to be one-directional, i.e., once blown to logic 1, it cannot be reset to logic 0.

### Demonstrative Examples

<p class="subheading">Example 1</p>
<br/>

<p><p>A chip implements a secure boot and uses the sensed value of a fuse 
    "do_secure_boot" to determine whether to perform a secure boot or not. If this fuse 
    value is "0", the system performs secure boot. Otherwise, it does not perform secure 
    boot.</p>
    <p>An attacker blows the "do_secure_boot" fuse to "1". After reset, the attacker loads a custom 
    bootloader, and, since the fuse value is now "1", the system does not perform secure boot, 
    and the attacker can execute their custom firmware image.</p>
    <p>Since by default, a fuse-configuration value is a "0", an attacker can blow it to a "1" with 
    inexpensive hardware.</p>
    <p>If the logic is reversed, an attacker cannot easily reset the fuse. Note that, with
    specialized and expensive equipment, an attacker with full physical access might be able to "unblow" the fuse
    value to a "0".</p>
</p>

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Logic should be designed in a way that blown fuses do not put the product into an insecure state that can be leveraged by an attacker. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design fuse logic so that the blown state (logic 1) always represents the more secure/restrictive configuration, making it impossible for an attacker to weaken security by blowing fuses. | Critical |
| Architecture and Design | Use multi-bit encoding schemes (e.g., Hamming distance checks or redundant fuse pairs) so that a single fuse blow cannot transition from a valid-secure to a valid-insecure state. | High |
| Implementation | Implement fuse integrity checks at boot that validate fuse patterns against expected secure encodings and halt boot if an inconsistent or attacked state is detected. | High |
| Testing | Perform post-silicon fuse manipulation testing where individual fuses are blown in isolation and in combination to verify no security downgrade is achievable. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
