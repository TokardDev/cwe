# CWE-1298 - Hardware Logic Contains Race Conditions
## CWE Explanation

This weakness occurs when the output of a hardware logic circuit depends on the timing or ordering of events that are not properly synchronized, leading to a race condition. In hardware, race conditions can manifest in two primary forms: combinational logic glitches and sequential/architectural races.

A **combinational glitch** happens when inputs to a logic gate arrive at slightly different times due to propagation delays. This can cause the output to momentarily flick to an incorrect state (a "glitch") before settling. If this output drives a sensitive control signal (like a "Grant Access" line or a clock input to a flip-flop), the glitch might be interpreted as a valid pulse, triggering an unauthorized action.

An **architectural race** (often a Time-of-Check to Time-of-Use issue) occurs when a hardware module checks a condition (like an address permission) at one point in time but uses the data at a later point, without ensuring the data hasn't changed in between. If the inputs are not latched or locked, an attacker can modify the signals after the check passes but before the operation completes, bypassing the security control.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | Attackers can bypass access control checks by changing transaction parameters after they are validated but before they are used. |
| **Integrity** | Race conditions can lead to data corruption or the execution of logic in an incorrect state. |
| **Availability** | Glitches on reset or clock lines can cause spurious resets or system hangs. |

## Attack Scenarios

### Scenario 1: DMA Input Race (TOCTOU)
A Direct Memory Access (DMA) controller is designed to move data between memory regions. It has an input port for the target address (`dest_addr`) and an input for the security configuration (`permissions`). The DMA state machine first checks if the `permissions` allow writing to `dest_addr`. If valid, it proceeds to the "Transfer" state. However, the DMA controller reads the `dest_addr` input directly from the bus during the transfer phase rather than using a latched copy captured during the check phase.

An attacker initiates a transfer to a valid, low-privileged address. As soon as the DMA controller validates the access and enters the "Transfer" state, the attacker toggles the `dest_addr` input signal to point to a restricted memory region (e.g., kernel memory). Because the DMA logic blindly uses the live input signal, it writes data to the restricted address, bypassing the initial permission check.

### Scenario 2: Combinational Logic Glitch
A security monitor generates a `RESET_SYSTEM` signal based on several inputs. The logic equation is implemented such that `RESET_SYSTEM = (A AND !B) OR (B AND C)`. Due to physical routing differences, a change in signal `B` arrives at the `AND` gate slightly later than the `OR` gate. When the system transitions from a safe state to another safe state, this timing mismatch causes the `RESET_SYSTEM` output to glitch high for a few nanoseconds. Although the logic indicates the output should remain low, the glitch is wide enough to trigger the asynchronous reset pin of the main processor, causing a Denial of Service.

## MITRE Reference Data

### Description

A race condition in the hardware logic results in undermining security guarantees of the system.

### Extended Description

A race condition in logic circuits typically occurs when a logic gate gets inputs from signals that have traversed different paths while originating from the same source. Such inputs to the gate can change at slightly different times in response to a change in the source signal. This results in a timing error or a glitch (temporary or permanent) that causes the output to change to an unwanted state before settling back to the desired state. If such timing errors occur in access control logic or finite state machines that are implemented in security sensitive flows, an attacker might exploit them to circumvent existing protections.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Adopting design practices that encourage designers to recognize and eliminate race conditions, such as Karnaugh maps, could result in the decrease in occurrences of race conditions. | - |
| Implementation | - | Logic redundancy can be implemented along security critical paths to prevent race conditions. To avoid metastability, it is a good practice in general to default to a secure state in which access is not given to untrusted agents. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Latch all security-critical input signals (addresses, permissions, configuration) at the start of a transaction and use only the latched copies throughout the operation, preventing TOCTOU races. | Critical |
| Implementation | Add consensus or redundancy terms to combinational logic on security-critical paths (e.g., include the `(A AND B)` hazard cover term) to eliminate transient glitches. | High |
| Verification | Run formal verification with temporal properties to prove that no security-critical output can glitch or change state outside of valid clock edges under any input transition sequence. | High |
| Testing | Use gate-level simulation with back-annotated delays to detect glitches on security-sensitive signals and validate that no spurious pulses can propagate to control logic. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
