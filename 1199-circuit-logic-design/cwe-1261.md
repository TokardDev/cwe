## CWE Explanation

This weakness occurs when hardware logic fails to anticipate or mitigate Single Event Upsets (SEUs)—spontaneous bit flips caused by external factors like cosmic radiation, alpha particles, or electromagnetic interference. While often considered a reliability issue, SEUs have significant security implications. If a bit flip occurs in a register holding sensitive state information (e.g., a "lock" bit, a privilege level bit, or an encryption key), it can instantaneously alter the system's security posture.

Hardware designers must implement resilience mechanisms, such as Error Correction Codes (ECC), redundancy (e.g., Triple Modular Redundancy), or safe state recovery logic, especially for security-critical storage elements. A failure to do so allows random environmental noise—or intentional fault injection attacks—to bypass security controls, corrupt sensitive data, or force the system into a denial-of-service state by triggering unrecoverable errors.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Access Control** | A bit flip in a control register could change a "User Mode" flag to "Supervisor Mode" or unlock a protected memory region. |
| **Availability** | Unhandled SEUs can cause system hangs, resets, or transitions to "safe modes" that disable essential functionality. |
| **Integrity** | Data integrity is compromised when bit flips modify stored values, instructions, or configuration parameters without detection. |

## Attack Scenarios

### Scenario 1: Privilege Escalation
A secure processor uses a single-bit register to store the current privilege level (`0` for User, `1` for Root). The hardware design relies on this single flip-flop without redundancy or error checking. An attacker, operating in a high-altitude environment or using a laser fault injection technique, induces a bit flip in this specific register. The value transitions from `0` to `1`. The hardware immediately begins executing the attacker's user-mode code with Root privileges, allowing full system compromise.

### Scenario 2: Denial of Service
A network router buffers packets in internal SRAM. To save area, the designers protected the SRAM with simple parity checks rather than Error Correction Codes (ECC). A cosmic ray strikes the chip, flipping a bit in a packet header. When the hardware reads the packet, the parity check fails. Instead of discarding the packet or correcting the error, the unrecoverable parity error triggers a non-maskable interrupt that resets the entire router. This results in a temporary but critical denial of service for the network.

## MITRE Reference Data

### Description

The hardware logic does not effectively handle when single-event upsets (SEUs) occur.

### Extended Description

Technology trends such as CMOS-transistor down-sizing, use of new materials, and system-on-chip architectures continue to increase the sensitivity of systems to soft errors. These errors are random, and their causes might be internal (e.g., interconnect coupling) or external (e.g., cosmic radiation). These soft errors are not permanent in nature and cause temporary bit flips known as single-event upsets (SEUs). SEUs are induced errors in circuits caused when charged particles lose energy by ionizing the medium through which they pass, leaving behind a wake of electron-hole pairs that cause temporary failures. If these failures occur in security-sensitive modules in a chip, it might compromise the security guarantees of the chip. For instance, these temporary failures could be bit flips that change the privilege of a regular user to root.

| Architecture and Design | - | Implement triple-modular redundancy around security-sensitive modules. | - |
| Architecture and Design | - | SEUs mostly affect SRAMs. For SRAMs storing security-critical data, implement Error-Correcting-Codes (ECC) and Address Interleaving. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement Triple Modular Redundancy (TMR) for all security-critical registers (privilege bits, lock bits, key storage control) so that a single bit flip cannot alter security state. | Critical |
| Architecture and Design | Deploy Error-Correcting Codes (ECC) with SECDED (Single Error Correction, Double Error Detection) on all SRAMs that store security-sensitive data such as keys, configuration, or access control policies. | Critical |
| Architecture and Design | Use address interleaving for security-critical SRAM regions to reduce the probability that a single particle strike corrupts multiple related bits. | High |
| Implementation | Add periodic scrubbing logic that reads, checks, and corrects ECC-protected memory at regular intervals to prevent accumulation of uncorrected soft errors. | High |
| Testing | Perform radiation testing or fault injection campaigns targeting security-critical flip-flops and SRAM cells to validate that TMR voting and ECC correction operate correctly under SEU conditions. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
