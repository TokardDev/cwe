## CWE Explanation

This weakness occurs when a hardware design implements a "lock bit" mechanism to protect critical configuration settings but fails to extend this protection to all related registers that influence the same security function. In many designs, a single lock bit is intended to freeze a set of security parameters after boot. However, if designers overlook auxiliary registers—such as enable bits, calibration values, or alternative access paths—an attacker can modify these unprotected registers to undermine the security mechanism, even if the "primary" registers are successfully locked.

For example, a secure boot configuration might lock the "root key hash" register, but leave the "secure boot enable" bit unlocking. An attacker wouldn't need to change the key hash; they could simply disable secure boot entirely. The root cause is typically a gap in the specification or verification of the lock signal's coverage, leaving side-doors open for reconfiguration after the trusted initialization phase.

| Aspect | Impact |
|--------|--------|
| **Access Control** | Attackers can bypass access controls by modifying unprotected configuration bits that disable or alter the behavior of the security mechanism. |
| **Integrity** | The integrity of the system's security state is compromised, as the hardware behavior no longer matches the "locked" configuration. |
| **Availability** | Attackers might be able to disable thermal protections or other safety mechanisms if their control registers are not properly included in the lock group. |

## Attack Scenario

A System-on-Chip (SoC) features a hardware thermal management unit designed to shut down the system if the temperature exceeds a critical threshold. To prevent malware from overheating the device, the "Critical Temperature Limit" register is protected by a write-once lock bit set by the trusted bootloader. However, the hardware designers forgot to include the "Thermal Shutdown Enable" bit or the "Temperature Sensor Calibration" register in the same lock protection domain.

An attacker, wishing to physically damage the device or cause a denial of service via overheating, gains kernel privileges. They find that while they cannot change the Critical Temperature Limit (because it is locked), they can simply write to the "Thermal Shutdown Enable" register to turn off the protection mechanism entirely. Alternatively, they could modify the calibration values so the sensor reports a low temperature regardless of the actual heat. By manipulating these unprotected side-channels, the attacker bypasses the intended safety lock.

## MITRE Reference Data

### Description

The product uses a register lock bit protection mechanism, but it does not ensure that the lock bit prevents modification of system registers or controls that perform changes to important hardware system configuration.

### Extended Description

Integrated circuits and hardware intellectual properties (IPs) might provide device configuration controls that need to be programmed after device power reset by a trusted firmware or software module, commonly set by BIOS/bootloader. After reset, there can be an expectation that the controls cannot be used to perform any further modification. This behavior is commonly implemented using a trusted lock bit, which can be set to disable writes to a protected set of registers or address regions. The lock protection is intended to prevent modification of certain system configuration (e.g., memory/memory protection unit configuration).

However, if the lock bit does not effectively write-protect all system registers or controls that could modify the protected system configuration, then an adversary may be able to use software to access the registers/controls and modify the protected hardware configuration.

<p>In this example note that only the CRITICAL_TEMP_LIMIT register is protected by the TEMP_SENSOR_LOCK bit, while the security design intent is to protect any modification of the critical temperature detection and response. 
</p>
<p>The response of the system, if the system heats to a critical temperature, is controlled by TEMP_HW_SHUTDOWN bit [1], which is not lockable. Also, the TEMP_SENSOR_CALIB register is not protected by the lock bit.
</p>
<p>By modifying the temperature sensor calibration, the conversion of the sensor data to a degree centigrade can be changed, such that the current temperature will never be detected to exceed critical temperature value programmed by the protected lock.
</p>
<p>Similarly, by modifying the TEMP_HW_SHUTDOWN.Enable bit, the system response detection of the current temperature exceeding critical temperature can be disabled.
