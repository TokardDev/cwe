# CWE-1254 - Incorrect Comparison Logic Granularity


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2019-10482](https://www.cve.org/CVERecord?id=CVE-2019-10482) | Smartphone OS uses comparison functions that are not in constant time, allowing side channels |
| [CVE-2019-10071](https://www.cve.org/CVERecord?id=CVE-2019-10071) | Java-oriented framework compares HMAC signatures  using  String.equals() instead of a constant-time algorithm, causing timing discrepancies |
| [CVE-2014-0984](https://www.cve.org/CVERecord?id=CVE-2014-0984) | Password-checking function in router terminates validation of a password entry when it encounters the first incorrect character, which allows remote attackers to obtain passwords via a brute-force attack that relies on timing differences in responses to incorrect password guesses, aka a timing side-channel attack. |

## CWE Explanation

This weakness involves the implementation of comparison logic—often used for verifying passwords, cryptographic signatures (MACs), or authorization tokens—in a way that reveals information about the secret inputs through timing side-channels. Secure comparison logic should execute in constant time, meaning the duration of the operation is independent of the input values.

However, if the hardware comparison logic is implemented with "early exit" optimizations (e.g., stopping the comparison as soon as the first mismatched byte is found), the time taken to reject an incorrect input correlates directly with how many leading bytes matched the secret. An attacker can exploit this by measuring the response time for various inputs. If a guess takes slightly longer to be rejected, it implies the first byte was correct. The attacker can then iteratively guess the second byte, and so on, reducing the complexity of a brute-force attack from exponential to linear (byte-by-byte recovery).

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Attackers can infer secret values (passwords, keys, tokens) by analyzing the timing of comparison operations. |
| **Access Control** | By recovering authentication secrets, attackers can bypass access controls and gain unauthorized privileges. |

## Attack Scenario

A hardware security module implements a password verification function using a byte-by-byte comparison loop. To optimize performance, the hardware logic is designed to terminate the loop and assert the "Access Denied" signal immediately upon detecting a mismatch. An attacker connects to the module and attempts to authenticate. They send a series of password guesses, using high-precision timers to measure the delay between sending the password and receiving the denial signal.

The attacker starts by guessing the first byte. Most guesses return the denial signal in $T$ cycles. However, when they guess the correct first byte, the hardware proceeds to compare the second byte before failing, resulting in a response time of $T + \Delta$ cycles. Recognizing this timing difference, the attacker fixes the first byte and proceeds to guess the second. This process repeats until the entire password is recovered, bypassing the security of the comparison logic.

## MITRE Reference Data

### Description

The product's comparison logic is performed over a series of steps rather than across the entire string in one operation. If there is a comparison logic failure on one of these steps, the operation may be vulnerable to a timing attack that can result in the interception of the process for nefarious purposes.

### Extended Description

Comparison logic is used to compare a variety of objects including passwords, Message Authentication Codes (MACs), and responses to verification challenges. When comparison logic is implemented at a finer granularity (e.g., byte-by-byte comparison) and breaks in the case of a comparison failure, an attacker can exploit this implementation to identify when exactly the failure occurred. With multiple attempts, the attacker may be able to guesses the correct password/response to challenge and elevate their privileges.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Implementation | Implement all security-critical comparisons (passwords, MACs, tokens) as constant-time operations: XOR all bytes, OR the results, and check the final aggregate, ensuring execution time is independent of input values. | Critical |
| Architecture and Design | Use dedicated hardware comparison blocks for authentication that are designed and verified to have fixed-latency execution paths with no early-exit optimization. | Critical |
| Verification | Add timing assertion checks in simulation that measure comparison logic latency across all input combinations and flag any variation exceeding one clock cycle. | High |
| Testing | In post-silicon, use high-precision timing measurements to validate that comparison operations show no statistically significant timing variation across correct and incorrect inputs. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
