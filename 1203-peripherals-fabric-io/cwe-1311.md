# CWE-1311 - Improper Translation of Security Attributes by Fabric Bridge

## CWE Explanation

System-on-Chip (SoC) designs often integrate Intellectual Property (IP) blocks that utilize different bus protocols (e.g., OCP, AHB, AXI). To facilitate communication between these disparate protocols, fabric bridges are employed. These protocols typically include signals that carry security attributes, such as the identity of the transaction initiator, privilege levels, and transaction types (read/write, secure/non-secure).

This weakness arises when a fabric bridge fails to correctly translate these security attribute signals when converting from one protocol to another. For instance, a "secure" or "privileged" bit in the source protocol might be mapped to a "non-secure" or "unprivileged" bit in the destination protocol, or vice versa.

Such mistranslations can have severe security implications. If an untrusted initiator's identity is translated to that of a trusted entity, the untrusted agent could bypass access controls, accessing sensitive memory or peripherals it should not be able to reach. Conversely, incorrect translation could lead to denial of service if legitimate trusted traffic is demoted and blocked.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read memory or access data belonging to trusted agents due to identity spoofing. |
| **Integrity** | The attacker can modify memory or send unauthorized commands to peripherals by assuming a trusted identity. |
| **Availability** | Low impact (*Denial of service is possible if trusted traffic is incorrectly downgraded and blocked*). |

## Attack Scenario

An attacker controls a non-privileged IP block connected to the system fabric via a bridge. The attacker initiates a transaction targeting a protected asset (like a sensitive register or memory region) that requires high privileges. The bridge, due to a design or implementation flaw in its translation logic, incorrectly maps the attacker's "untrusted" security attribute to a "trusted" attribute on the destination bus side. As a result, the transaction arrives at the target with privileged status, bypassing the access control check. The attacker successfully reads or modifies the protected asset.

## MITRE Reference Data

### Description

The bridge incorrectly translates security attributes from either trusted to untrusted or from untrusted to trusted when converting from one fabric protocol to another.

### Extended Description

A bridge allows IP blocks supporting different fabric protocols to be integrated into the system.  Fabric end-points or interfaces usually have dedicated signals to transport security attributes. For example, HPROT signals in AHB, AxPROT signals in AXI, and MReqInfo and SRespInfo signals in OCP.

The values on these signals are used to indicate the security attributes of the transaction. These include the immutable hardware identity of the controller initiating the transaction, privilege level, and type of transaction (e.g., read/write, cacheable/non-cacheable, posted/non-posted).

A weakness can arise if the bridge IP block, which translates the signals from the protocol used in the IP block endpoint to the protocol used by the central bus, does not properly translate the security attributes. As a result, the identity of the initiator could be translated from untrusted to trusted or vice-versa. This could result in access-control bypass, privilege escalation, or denial of service.

### Demonstrative Examples

#### Example 1

The bridge interfaces between OCP and AHB end points. OCP uses MReqInfo signal to indicate security attributes, whereas AHB uses HPROT signal to indicate the security attributes. The width of MReqInfo can be customized as needed. In this example, MReqInfo is 5-bits wide and carries the privilege level of the OCP controller.

The values 5'h11, 5'h10, 5'h0F, 5'h0D, 5'h0C, 5'h0B, 5'h09, 5'h08, 5'h04, and 5'h02 in MReqInfo indicate that the request is coming from a privileged state of the OCP bus controller. Values 5'h1F, 5'h0E, and 5'h00 indicate untrusted, privilege state.

Though HPROT is a 5-bit signal, we only consider the lower, two bits in this example. HPROT values 2'b00 and 2'b10 are considered trusted, and 2'b01 and 2'b11 are considered untrusted.

The OCP2AHB bridge is expected to translate trusted identities on the controller side to trusted identities on the responder side.  Similarly, it is expected to translate untrusted identities on the controller side to untrusted identities on the responder side.

```verilog
module ocp2ahb
(
  ahb_hprot,
  ocp_mreqinfo
);

output [1:0] ahb_hprot;        // output is 2 bit signal for AHB HPROT
input [4:0] ocp_mreqinfo;      // input is 5 bit signal from OCP MReqInfo
wire [6:0] p0_mreqinfo_o_temp; // OCP signal that transmits hardware identity of bus controller

wire y;

reg [1:0] ahb_hprot;

// hardware identity of bus controller is in bits 5:1 of p0_mreqinfo_o_temp signal
assign p0_mreqinfo_o_temp[6:0] = {1'b0, ocp_mreqinfo[4:0], y};

always @*
begin
  case (p0_mreqinfo_o_temp[4:2])
    000: ahb_hprot = 2'b11;    // OCP MReqInfo to AHB HPROT mapping
    001: ahb_hprot = 2'b00;
    010: ahb_hprot = 2'b00;
    011: ahb_hprot = 2'b01;
    100: ahb_hprot = 2'b00;
    101: ahb_hprot = 2'b00;
    110: ahb_hprot = 2'b10;
    111: ahb_hprot = 2'b00;
  endcase
end
endmodule
```

Logic in the case statement only checks for MReqInfo bits 4:2, i.e., hardware-identity bits 3:1. When ocp_mreqinfo is 5'h1F or 5'h0E, p0_mreqinfo_o_temp[2] will be 1. As a result, untrusted IDs from OCP 5'h1F and 5'h0E get translated to trusted ahb_hprot values 2'b00.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | The translation must map signals in such a way that untrusted agents cannot map to trusted agents or vice-versa. | - |
| Implementation | - | Ensure that the translation maps signals in such a way that untrusted agents cannot map to trusted agents or vice-versa. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Define a complete and exhaustive mapping table for all security attribute signals between source and destination protocols, covering every possible input value. | Critical |
| Architecture and Design | Implement a default-deny policy in bridge translation logic: any unmapped or unknown security attribute value must be translated to the least-privileged/untrusted state. | Critical |
| Implementation | Add assertion-based verification in RTL to flag any case where an untrusted source identity is translated to a trusted destination identity. | High |
| Testing | Perform exhaustive pre-silicon simulation of all MReqInfo-to-HPROT (or equivalent) mappings, including boundary and corner-case values. | High |
| Testing | Include fabric bridge security-attribute translation tests in post-silicon validation plans, injecting transactions from untrusted masters and verifying they remain untrusted after bridge translation. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
