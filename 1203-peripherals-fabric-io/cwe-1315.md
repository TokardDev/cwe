# CWE-1315 - Improper Setting of Bus Controlling Capability in Fabric End-point

## CWE Explanation

In modern System-on-Chip (SoC) architectures, the on-chip fabric connects various components, some acting as "controllers" (or masters) that initiate transactions, and others as "responders" (or slaves) that service them. This weakness occurs when a fabric end-point, intended solely as a responder, has its bus controlling capability enabled. This is often controlled by a configurable register bit in the endpoint's interface (e.g., PCIe).

If this capability is not disabled by default in hardware or is not securely disabled by early firmware, a malicious or compromised peripheral device can initiate its own transactions on the bus. This violates the principle of least privilege.

A responder device with bus mastering capabilities can read or write to any memory address or peripheral in the system that isn't explicitly protected against it. This allows a low-trust peripheral (like an audio codec or a WiFi module) to completely subvert the system's security model.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read sensitive memory or data from other peripherals. |
| **Integrity** | The attacker can modify system memory, configuration registers, or data belonging to other processes. |
| **Availability** | The attacker can disrupt system operations, cause crashes, or reset the system. |

## Attack Scenario

An attacker compromises a peripheral device (e.g., an audio codec) that is connected to the system fabric. This device is designed to be a responder, only replying to requests from the main CPU. However, the "Bus Master Enable" bit in its configuration register was left enabled by default or was not disabled by the boot firmware. The attacker uses the compromised peripheral to initiate Direct Memory Access (DMA) write transactions targeting the system's kernel memory. Since the fabric sees the device as a valid bus master, the transactions are allowed, enabling the attacker to overwrite kernel code and gain full control of the host system.

## MITRE Reference Data

### Description

The bus controller enables bits in the fabric end-point to allow responder devices to control transactions on the fabric.

### Extended Description

To support reusability, certain fabric interfaces and end points provide a configurable register bit that allows IP blocks connected to the controller to access other peripherals connected to the fabric. This allows the end point to be used with devices that function as a controller or responder. If this bit is set by default in hardware, or if firmware incorrectly sets it later, a device intended to be a responder on a fabric is now capable of controlling transactions to other devices and might compromise system security.

### Demonstrative Examples

#### Example 1

A typical, phone platform consists of the main, compute core or CPU, a DRAM-memory chip, an audio codec, a baseband modem, a power-management-integrated circuit ("PMIC"), a connectivity (WiFi and Bluetooth) modem, and several other analog/RF components. The main CPU is the only component that can control transactions, and all the other components are responder-only devices. All the components implement a PCIe end-point to interface with the rest of the platform. The responder devices should have the bus-control-enable bit in the PCIe-end-point register set to 0 in hardware to prevent the devices from controlling transactions to the CPU or other peripherals.

The audio-codec chip does not have the bus-controller-enable-register bit hardcoded to 0.  There is no platform-firmware flow to verify that the bus-controller-enable bit is set to 0 in all responders.

Audio codec can now master transactions to the CPU and other platform components. Potentially, it can modify assets in other platform components to subvert system security.

Platform firmware includes a flow to check the configuration of bus-controller-enable bit in all responder devices. If this register bit is set on any of the responders, platform firmware sets it to 0. Ideally, the default value of this register bit should be hardcoded to 0 in RTL. It should also have access control to prevent untrusted entities from setting this bit to become bus controllers.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Hardcode the bus-master-enable bit to 0 in RTL for all IP blocks that are designed exclusively as responders, making the bit read-only. | Critical |
| Architecture and Design | Implement an IOMMU or equivalent DMA firewall to restrict the address ranges accessible by any bus master, even if the capability is erroneously enabled. | Critical |
| Implementation | Add a secure-boot firmware step that enumerates all fabric endpoints and explicitly disables bus-master capability on all responder-only devices before enabling any untrusted peripherals. | High |
| Testing | Include pre-silicon and post-silicon test cases that verify responder-only devices cannot initiate transactions, both with default configuration and after attempted malicious writes to the bus-master-enable register. | High |
| System Configuration | Implement runtime monitoring or hardware watchdog that detects and alerts if a responder device unexpectedly initiates bus transactions. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
