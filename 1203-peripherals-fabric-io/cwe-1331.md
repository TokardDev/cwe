# CWE-1331 - Improper Isolation of Shared Resources in Network On Chip (NoC)


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2021-33096](https://www.cve.org/CVERecord?id=CVE-2021-33096) | Improper isolation of shared resource in a network-on-chip leads to denial of service |

## CWE Explanation

Network-on-Chip (NoC) architectures provide a scalable interconnect for multicore Systems-on-Chip (SoC), treating communication between modules (agents) as packet-based traffic. To optimize efficiency and area, NoCs often share internal resources such as buffers, crossbars, switches, and virtual channels among different traffic flows.

This weakness arises when the NoC fails to properly isolate these shared resources between trusted and untrusted agents. If resources are shared without strict temporal or spatial partitioning (e.g., priority arbitration or dedicated virtual channels), traffic from one domain can affect the timing and performance of traffic in another domain.

This interference creates a side-channel (specifically a timing channel). An untrusted agent can infer sensitive information about the activity of a trusted agent by observing contention, latency, or bandwidth fluctuations. For example, an attacker might deduce cryptographic keys by analyzing how a crypto-accelerator's memory access patterns affect the shared bus latency.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can infer sensitive data (like cryptographic keys) via timing side-channels. |
| **Integrity** | Low impact (*Integrity is generally not directly compromised by the side-channel itself, though DoS is possible*). |
| **Availability** | The attacker can cause denial of service (degraded throughput/latency) by saturating shared resources. |

## Attack Scenario

An attacker runs a malicious process on a non-secure processor core connected to the NoC. Simultaneously, a secure core performs a cryptographic operation (e.g., RSA) that involves data-dependent memory accesses. Both cores share a common link or router output port in the NoC mesh. The attacker's process generates a high volume of traffic and measures the latency of its own packets. Variations in this latency correlate with the contention caused by the secure core's traffic. By analyzing these timing variations over time, the attacker reconstructs the execution path of the RSA algorithm, eventually extracting the private key.

## MITRE Reference Data

### Description

The Network On Chip (NoC) does not isolate or incorrectly isolates its on-chip-fabric and internal resources such that they are shared between trusted and untrusted agents, creating timing channels.

### Extended Description

Typically, network on chips (NoC) have many internal resources that are shared between packets from different trust domains. These resources include internal buffers, crossbars and switches, individual ports, and channels. The sharing of resources causes contention and introduces interference between differently trusted domains, which poses a security threat via a timing channel, allowing attackers to infer data that belongs to a trusted agent. This may also result in introducing network interference, resulting in degraded throughput and latency.

### Demonstrative Examples

#### Example 1

Consider a NoC that implements a one-dimensional mesh network with four nodes. This supports two flows: Flow A from node 0 to node 3 (via node 1 and node 2) and Flow B from node 1 to node 2. Flows A and B share a common link between Node 1 and Node 2.  Only one flow can use the link in each cycle.

One of the masters to this NoC implements a cryptographic algorithm (RSA), and another master to the NoC is a core that can be exercised by an attacker. The RSA algorithm performs a modulo multiplication of two large numbers and depends on each bit of the secret key. The algorithm examines each bit in the secret key and only performs multiplication if the bit is 1. This algorithm is known to be prone to timing attacks. Whenever RSA performs multiplication, there is additional network traffic to the memory controller. One of the reasons for this is cache conflicts.

Since this is a one-dimensional mesh, only one flow can use the link in each cycle.  Also, packets from the attack program and the RSA program share the output port of the network-on-chip.  This contention results in network interference, and the throughput and latency of one flow can be affected by the other flow's demand.

The attacker runs a loop program on the core they control, and this causes a cache miss in every iteration for the RSA algorithm. Thus, by observing network-traffic bandwidth and timing, the attack program can determine when the RSA algorithm is doing a multiply operation (i.e., when the secret key bit is 1) and eventually extract the entire, secret key.

There may be different ways to fix this particular weakness.

Implement priority-based arbitration inside the NoC and have dedicated buffers or virtual channels for routing secret data from trusted agents.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Allocate dedicated virtual channels or physical links for trusted traffic flows, ensuring complete temporal and spatial isolation from untrusted traffic. | Critical |
| Architecture and Design | Implement constant-time or fixed-rate arbitration for security-sensitive flows to eliminate observable timing variations caused by contention. | High |
| Implementation | Configure NoC Quality-of-Service (QoS) policies to guarantee minimum bandwidth and maximum latency for trusted agents, preventing starvation by untrusted traffic. | High |
| Implementation | Partition shared buffers and crossbar resources so that untrusted agents cannot fill or monopolize resources used by trusted agents. | High |
| Testing | Perform side-channel analysis testing in pre-silicon simulation by measuring latency and bandwidth variations of trusted flows under adversarial traffic injection from untrusted agents. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
