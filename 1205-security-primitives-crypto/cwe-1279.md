# CWE-1279 - Cryptographic Operations are run Before Supporting Units are Ready

## CWE Explanation

This weakness occurs when a cryptographic hardware module or software routine initiates an operation before its necessary dependencies—such as a Random Number Generator (RNG), key management unit, or entropy source—are fully initialized and ready to provide valid data. In complex hardware systems, different blocks power up and initialize at different rates. If a cryptographic unit requests data (like a seed or a key) from a supporting unit that is still in a reset or initialization state, it may receive invalid, predictable, or default values (often all zeros or a fixed constant).

Using these non-ready values for cryptographic operations catastrophically weakens the security. For example, if an RNG returns zero because it hasn't gathered enough entropy yet, the "random" key generated will be zero. Attackers can exploit this by predicting the keys or nonces generated immediately after a system reset or power cycle.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | High (Predictable keys/seeds lead to decryption) |
| **Integrity** | High (Predictable nonces allow signature forgery) |
| **Availability** | Low |

## Attack Scenario

An attacker observes a cryptographic device immediately after it is powered on. The device's architecture includes a dedicated hardware Random Number Generator (RNG) used to seed the key generation process. However, the system software or control logic attempts to read from the RNG register before the RNG's internal entropy pool has sufficiently accumulated and the "ready" status bit is set. Consequently, the RNG hardware returns a default value of `0x00000000`. The key generation routine accepts this value as a valid random seed and generates a cryptographic key. The attacker, knowing that the device is vulnerable during the boot sequence, predicts the key derived from the zero seed and successfully decrypts the initial secure communications sent by the device.

## MITRE Reference Data

### Description

Performing cryptographic operations without ensuring that the supporting inputs are ready to supply valid data may compromise the cryptographic result.

### Extended Description

Many cryptographic hardware units depend upon other hardware units to supply information to them to produce a securely encrypted result. For example, a cryptographic unit that depends on an external random-number-generator (RNG) unit for entropy must wait until the RNG unit is producing random numbers. If a cryptographic unit retrieves a private encryption key from a fuse unit, the fuse unit must be up and running before a key may be supplied.
