# CWE-1431 - Driving Intermediate Cryptographic State/Results to Hardware Module Outputs
## CWE Explanation

This weakness occurs in hardware cryptographic modules when intermediate calculation results—which are internal states of the algorithm—are exposed on the module's output ports before the final ciphertext or signature is ready. Hardware modules typically take multiple clock cycles to complete a cryptographic operation (like AES or RSA). If the output port is wired directly to the internal state register without gating logic, the changing values of the register during the computation are visible to any component connected to that bus.

Even if a "valid" signal is provided to indicate when the final result is ready, an attacker (or a compromised peripheral on the same bus) can ignore the valid signal and sample the output lines at every clock cycle. These intermediate values often correlate directly with the secret key or the plaintext, allowing for cryptographic analysis (like algebraic attacks or correlation attacks) that is much simpler than breaking the full algorithm.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | High (Intermediate states leak key info or plaintext) |
| **Integrity** | Low |
| **Availability** | Low |

## Attack Scenario

An attacker has access to the system bus or a debug interface connected to a hardware encryption accelerator. The accelerator takes 10 clock cycles to encrypt a block of data using AES-128. The module includes a "done" interrupt to signal completion, but its 128-bit output data bus is continuously driven by the internal state register. In the next step, the attacker triggers an encryption operation and uses a logic analyzer or a compromised bus master to record the values on the output bus for each of the 10 clock cycles. By capturing the state of the AES rounds as they progress, the attacker obtains the output of the internal "SubBytes" and "ShiftRows" operations. Using these intermediate values, the attacker can mathematically derive the round keys and ultimately the master encryption key, bypassing the need for complex side-channel analysis.

## MITRE Reference Data

### Description

The product uses a hardware module implementing a cryptographic algorithm that writes sensitive information about the intermediate state or results of its cryptographic operations via one of its output wires (typically the output port containing the final result).

                <p><p>In line 50 above, data_state_q is assigned to data_o. Since data_state_q
			    contains intermediate state/results, this allows an attacker to obtain
			    these results through data_o.</p></p> 
                
                
                
                
                <p>
			      <p>In line 50 of the fixed logic below,
			      while &#34;data_state_q&#34; does not contain the final result, 
			      a &#34;sanitizing&#34; mechanism drives a safe default value (i.e., 0) to
			      &#34;data_o&#34; instead of the value of &#34;data_state_q&#34;.
			      In doing so, the mechanism prevents
			      the exposure of intermediate state/results which could be used to
			      break soundness of the cryptographic operation being performed. A
			      real-world example of this weakness and mitigation can be seen in a
			      pull request that was submitted to the OpenTitan Github repository
			      [<a href="#REF-1469_1431">REF-1469</a>].</p>
