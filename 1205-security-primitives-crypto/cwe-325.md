# CWE-325 - Missing Cryptographic Step


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2001-1585](https://www.cve.org/CVERecord?id=CVE-2001-1585) | Missing challenge-response step allows authentication bypass using public key. |

## CWE Explanation

This weakness occurs when a hardware or software implementation of a cryptographic algorithm omits a necessary step defined in the algorithm's specification. Cryptographic protocols are designed as complete systems where every operation—initialization, padding, iteration, finalization, and verification—plays a critical role in maintaining security properties.

When a step is skipped, often due to performance optimizations, complexity reduction, or misunderstanding of the specification, the resulting implementation fails to provide the expected security level. For example, omitting a salting step in hashing, skipping an initialization vector (IV) update, or failing to iterate over message blocks can render the cryptography trivially breakable or susceptible to specific attacks like replay or collision.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | High (Encryption may be weak or reversible) |
| **Integrity** | High (Signatures or MACs may be forged) |
| **Availability** | Low |

## Attack Scenario

An attacker is able to supply a long message (greater than the block size) to a hardware-based HMAC engine. The HMAC engine's implementation correctly processes the first block but fails to implement the required iterative hashing step for subsequent blocks, effectively ignoring the rest of the message or processing it incorrectly. In the next step, the attacker generates two different messages that share the same first block but differ in the subsequent blocks. Because the engine skips the iteration step for the extra data, it produces the same Message Authentication Code (HMAC) for both messages. The attacker exploits this collision to forge a valid signature for a malicious payload, bypassing the integrity check.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement the full cryptographic algorithm as specified in the standard (e.g., NIST FIPS 198-1 for HMAC, FIPS 197 for AES), including all iterative hashing, padding, and finalization steps without omissions. | Critical |
| Architecture and Design | For hardware HMAC/hash engines, support arbitrary message lengths by implementing a FIFO-based message ingestion pipeline with proper block iteration, not a fixed-width single-shot input. | Critical |
| Verification and Testing | Validate the crypto implementation against official Known Answer Test (KAT) vectors covering edge cases: empty messages, single-block messages, multi-block messages, and messages requiring padding. | High |
| Implementation | Use established open-source reference designs (e.g., OpenTitan HMAC) as a baseline for cryptographic IP implementation to avoid inadvertently skipping protocol-mandated steps. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
