# CWE-1329 - Reliance on Component That is Not Updateable

## CWE Explanation

This weakness occurs when a product architecture incorporates sub-components (hardware IPs, firmware blobs, or third-party libraries) that cannot be patched or updated once deployed. In modern system design, modularity is key, but if a sub-component lacks an update mechanism (due to being in ROM, having no write interface, or lacking a cryptographic update routine), any vulnerability discovered within that component becomes permanent.

Unlike CWE-1277 which focuses on the product as a whole, this CWE highlights the risk of relying on *specific dependencies* that are immutable. A system might be updateable, but if it relies on a "black box" crypto accelerator or a boot ROM that is frozen in silicon, the security of the entire system is anchored to the correctness of that immutable block. If a flaw is found there, the "updateable" main system cannot fix it.

The root cause often lies in supply chain choices, cost-cutting (ROM is cheaper than Flash), or security architecture decisions that prioritize immutability for integrity without considering remediation for bugs.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can exploit unpatchable bugs to extract secrets |
| **Integrity** | The attacker can compromise the component's function permanently |
| **Availability** | The attacker can disable the component or the entire system |

## Attack Scenarios

### Scenario 1
An attacker targets a smart home hub that uses a specific third-party microchip for handling network communications. While the hub's main operating system can be updated over-the-air (OTA), the network chip runs a fixed firmware image burned into Read-Only Memory (ROM) by the supplier. The attacker discovers a remote code execution vulnerability in the network chip's packet processing logic. Because the firmware is immutable, the manufacturer cannot issue a patch. The attacker uses this unfixable entry point to compromise every deployed hub.

### Scenario 2
A System-on-Chip (SoC) utilizes an immutable Root-of-Trust (RoT) stored in Mask ROM to verify the bootloader. A vulnerability is found in the RoT's signature verification routine that allows malformed signatures to pass validation. Since the code is physical hardware (Mask ROM) and cannot be altered, the RoT cannot be patched. Attackers leverage this to bypass Secure Boot permanently on all devices using this SoC revision.

### Scenario 3
A hardware module for JTAG debugging is protected by a password check. The password and the cryptographic key used to verify it are hardcoded into the silicon logic (netlist) rather than stored in a programmable register. An attacker reverse-engineers the chip or leaks the key. Because the key is part of the fixed hardware logic and not a mutable register, the vendor cannot "rotate" or update the key. The attacker gains permanent, unauthorized debug access to the device.

## MITRE Reference Data

### Description

The product contains a component that cannot be updated or patched in order to remove vulnerabilities or significant bugs.

### Extended Description

If the component is discovered to contain a vulnerability or critical bug, but the issue cannot be fixed using an update or patch, then the product's owner will not be able to protect against the issue. The only option might be replacement of the product, which could be too financially or operationally expensive for the product owner. As a result, the inability to patch or update can leave the product open to attacker exploitation or critical operation failures. This weakness can be especially difficult to manage when using ROM, firmware, or similar components that traditionally have had limited or no update capabilities.

In industries such as healthcare, "legacy" devices can be operated for decades. As a US task force report [REF-1197] notes, "the inability to update or replace equipment has both large and small health care delivery organizations struggle with numerous unsupported legacy systems that cannot easily be replaced (hardware, software, and operating systems) with large numbers of vulnerabilities and few modern countermeasures."

While hardware can be prone to this weakness, software systems can also be affected, such as when a third-party driver or library is no longer actively maintained or supported but is still critical for the required functionality.

#### Example 2

A System-on-Chip (SOC) implements a Root-of-Trust (RoT) in ROM to boot secure code. However, at times this ROM code might have security vulnerabilities and need to be patched. Since ROM is immutable, it can be impossible to patch.

ROM does not have built-in application-programming interfaces (APIs) to patch if the code is vulnerable. Implement mechanisms to patch the vulnerable ROM code.

#### Example 3

The example code is taken from the JTAG module of the buggy OpenPiton SoC of HACK@DAC'21. JTAG is protected with a password checker. Access to JTAG operations will be denied unless the correct password is provided by the user. This user-provided password is first sent to the HMAC module where it is hashed with a secret crypto key. This user password hash (pass_hash) is then compared with the hash of the correct password (exp_hash). If they match, JTAG will then be unlocked.

```verilog
   /* (bad code) Example Language: Verilog */
   module dmi_jtag(...)(...);
   ...
      PassChkValid: begin
      if(hashValid) begin
         if(exp_hash == pass_hash) begin
            pass_check = 1'b1;
         end else begin
            pass_check = 1'b0;
         end
         state_d = Idle;
      end else begin
      state_d = PassChkValid;
      end
      end
   ...
      hmac hmac(
   ...
      .key_i(256'h24e6fa2254c2ff632a41b...),
   ...
      );
   ...
   endmodule
```

However, the SoC's crypto key is hardcoded into the design and cannot be updated [REF-1387]. Therefore, if the key is leaked somehow, there is no way to reprovision the key without having the device replaced.

To fix this issue, a local register should be used (hmac_key_reg) to store the crypto key. If designers need to update the key, they can upload the new key through an input port (hmac_key_i) to the local register by enabling the patching signal (hmac_patch_en) [REF-1388].

```verilog
   /* (good code) Example Language: Verilog */
   module dmi_jtag(...
   ) (
      input logic [255:0] hmac_key_i,
      input logic         hmac_patch_en,
      ... 
      reg [255:0] hmac_key_reg;
      ...
   );
   ...
      always_ff @(posedge tck_i or negedge trst_ni) begin
      ...
      if (hmac_patch_en)
         hmac_key_reg <= hmac_key_i;
      ...
      end
   ...
      hmac hmac(
      ...
      .key_i(hmac_key_reg),
      ...
      );
   ...
   endmodule
```
