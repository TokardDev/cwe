## CWE Explanation

This weakness occurs when a hardware component fails to completely remove sensitive data when intended, leaving residual traces that can be recovered. In hardware, data "erasure" is often more complex than in software. Simply declaring memory as "free" or overwriting it once might not physically remove the data due to phenomena like data remanence in magnetic media, residual charge in volatile memory (cold boot attacks), or wear-leveling algorithms in flash memory that keep old copies of data in invalid pages.

If the data removal process (e.g., sanitization, factory reset) does not account for these physical properties, an attacker with physical access or specialized equipment can recover the "deleted" information. This is critical for devices that are decommissioned, returned for service, or repurposed.

The root cause is a mismatch between the logical view of data deletion (software/firmware command) and the physical reality of how the storage medium retains information.

## Attack Scenario

An attacker acquires a used hardware security module (HSM) that was decommissioned by a bank. The previous owner had issued a "factory reset" command which logically cleared the encryption keys from the active memory map. However, the reset process did not trigger a secure purge (such as multiple overwrites or a specific hardware erase voltage) of the underlying flash cells. The attacker desolders the memory chip and uses a chip programmer to read the raw physical memory. By analyzing the physical state of the cells (or accessing pages marked as invalid by the flash controller), the attacker successfully recovers the master encryption key used by the previous owner.

## MITRE Reference Data

### Description

The product's data removal process does not completely delete all data and potentially sensitive information within hardware components.

### Extended Description

Physical properties of hardware devices, such as remanence of magnetic media, residual charge of ROMs/RAMs, or screen burn-in may still retain sensitive data after a data removal process has taken place and power is removed.

Recovering data after erasure or overwriting is possible due to a phenomenon called data remanence. For example, if the same value is written repeatedly to a memory location, the corresponding memory cells can become physically altered to a degree such that even after the original data is erased that data can still be recovered through physical characterization of the memory cells.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement hardware-enforced secure erase that overwrites all physical memory cells (including wear-leveled spare pages) with multiple patterns before declaring erasure complete. | Critical |
| Architecture and Design | Use self-encrypting storage with cryptographic erase capability, where destroying the encryption key renders all stored data irrecoverable. | Critical |
| Implementation | Program factory-reset and decommissioning routines to invoke the storage controller's secure-erase command rather than relying on file-system-level deletion. | High |
| Testing | Validate data removal effectiveness by performing physical memory dumps after erase operations to confirm no residual sensitive data remains recoverable. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
