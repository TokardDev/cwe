## CWE Explanation

This weakness addresses the supply chain and architectural risk of integrating components (hardware IPs, software libraries, third-party modules) that do not meet the security, reliability, or trustworthiness standards required for the overall product. In modern development, reusing existing components is essential for speed and cost-efficiency. However, if a component is sourced from a vendor with poor security practices, lacks transparency (e.g., no SBOM), or is known to be vulnerable/malicious, relying on it compromises the security of the entire system.

"Trustworthiness" here implies not just the absence of malware, but also the component's maintainability, the vendor's responsiveness to vulnerability disclosures, and the component's resilience against attacks. If a critical security function (like encryption or authentication) is delegated to an untrustworthy component, the system's security guarantees are effectively nullified.

The root cause is often insufficient due diligence during the procurement or design phase, prioritizing cost over security, or lack of visibility into the sub-components of a purchased module.

## Attack Scenario

A defense contractor designs a secure communication device but sources the Wi-Fi connectivity module from a third-party vendor to save costs. The vendor provides the module as a "black box" with no source code and limited documentation. Unknown to the contractor, the Wi-Fi module runs an outdated OS with known unpatchable vulnerabilities and contains a hardcoded debug backdoor used by the vendor for factory testing. An attacker on the same network identifies the module's MAC address OUI, exploits the known vulnerability in the outdated OS, and uses the backdoor to pivot from the Wi-Fi module into the main secure processor, decrypting and exfiltrating classified communications.

## MITRE Reference Data

### Description

The product is built from multiple separate components, but it uses a component that is not sufficiently trusted to meet expectations for security, reliability, updateability, and maintainability.

### Extended Description

Many modern hardware and software products are built by combining multiple smaller components together into one larger entity, often during the design or architecture phase. For example, a hardware component might be built by a separate supplier, or the product might use an open-source software library from a third party.

Regardless of the source, each component should be sufficiently trusted to ensure correct, secure operation of the product. If a component is not trustworthy, it can produce significant risks for the overall product, such as vulnerabilities that cannot be patched fast enough (if at all); hidden functionality such as malware; inability to update or replace the component if needed for security purposes; hardware components built from parts that do not meet specifications in ways that can lead to weaknesses; etc. Note that a component might not be trustworthy even if it is owned by the product vendor, such as a software component whose source code is lost and was built by developers who left the company, or a component that was developed by a separate company that was acquired and brought into the product's own company.

Note that there can be disagreement as to whether a component is sufficiently trustworthy, since trust is ultimately subjective. Different stakeholders (e.g., customers, vendors, governments) have various threat models and ways to assess trust, and design/architecture choices might make tradeoffs between security, reliability, safety, privacy, cost, and other characteristics.
