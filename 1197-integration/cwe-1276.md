# CWE-1276 - Hardware Child Block Incorrectly Connected to Parent System

## CWE Explanation

This weakness occurs during the system integration phase when a hardware Intellectual Property (IP) block is instantiated within a larger parent design (such as a System-on-Chip). While the internal logic of the IP block may be secure, its security guarantees rely on correct connections to the external environment. If signals—such as resets, interrupts, privilege indicators, or debug enables—are incorrectly connected, left floating, or hardwired to inappropriate logic levels, the IP's security mechanisms can be bypassed or rendered ineffective.

For example, a security-critical input might be accidentally tied to a fixed "safe" value, preventing the IP from ever entering a restricted state, or an interrupt line intended to trigger data scrubbing might be left unconnected. These integration errors often do not break basic functionality, making them difficult to detect during standard functional verification, yet they introduce significant security vulnerabilities like privilege escalation or information leakage.

The root cause is typically a mismatch between the IP's interface requirements and the integrator's implementation, often due to manual connection errors, ambiguous documentation, or copy-paste mistakes in Hardware Description Language (HDL) code.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can access sensitive information that should have been flushed or protected by privilege checks. |
| **Integrity** | The attacker can modify protected data or configuration registers by bypassing access controls. |
| **Availability** | Low impact (*System instability or inconsistent states may occur if resets or interrupts fail to propagate*). |

## Attack Scenarios

### Scenario 1
An attacker sends data from a non-secure source to a TrustZone-aware peripheral. Due to an integration error, the "security level" input signal of the peripheral is permanently grounded (hardwired to a "secure" logic level) instead of being connected to the actual bus security signal. Consequently, the peripheral treats the attacker's non-secure data as secure, bypassing the intended privilege isolation and allowing the attacker to perform unauthorized operations.

### Scenario 2
An attacker triggers a system event, such as an interrupt, that is supposed to initiate a context switch and flush sensitive data from a Control Status Register (CSR) module. Due to an integration error, the interrupt signal is left unconnected (floating) at the CSR module interface. The module fails to detect the interrupt and does not flush the registers. As a result, the attacker (or a process in a lower privilege context) can read the stale sensitive information remaining in the registers.

## MITRE Reference Data

### Description
Signals between a hardware IP and the parent system design are incorrectly connected causing security risks.

### Extended Description
Individual hardware IP must communicate with the parent system in order for the product to function correctly and as intended. If implemented incorrectly, while not causing any apparent functional issues, may cause security issues. For example, if the IP should only be reset by a system-wide hard reset, but instead the reset input is connected to a software-triggered debug mode reset (which is also asserted during a hard reset), integrity of data inside the IP can be violated.

<p>Many SoCs use hardware to partition system resources between trusted and untrusted entities. One example of this concept is the Arm TrustZone, in which the processor and all security-aware IP attempt to isolate resources based on the status of a privilege bit. This privilege bit is part of the input interface in all TrustZone-aware IP. If this privilege bit is accidentally grounded or left unconnected when the IP is instantiated, privilege escalation of all input data may occur.</p>

<p>In the above example from HACK@DAC&#39;21, since interrupt signals are not properly connected, the CSR module will fail to send notifications in the event of interrupts. Consequently, critical information in CSR registers that should be flushed or modified in response to an interrupt won&#39;t be updated. <p>To address the aforementioned vulnerability, developers must follow a two-step approach. First, they should ensure that all module signals are properly connected. This can often be facilitated using automated tools, and many simulators and sanitizer tools issue warnings when a signal remains unconnected or floats. Second, it is imperative to validate that the signals connected to a module align with the specifications. In the provided example, the developer should establish the correct connection of interrupt signals from the parent module (Ariane core) to the child module (csr_regfile) [<a href="#REF-1363_1276">REF-1363</a>].</p>
