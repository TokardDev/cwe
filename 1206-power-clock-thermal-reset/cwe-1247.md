# CWE-1247 - Improper Protection Against Voltage and Clock Glitches


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2019-17391](https://www.cve.org/CVERecord?id=CVE-2019-17391) | Lack of anti-glitch protections allows an attacker to launch a physical attack to bypass the secure boot and read protected eFuses. |
| [CVE-2021-33478](https://www.cve.org/CVERecord?id=CVE-2021-33478) | IP communication firmware allows access to a boot shell via certain impulses |

## CWE Explanation

This weakness occurs when hardware devices lack proper circuitry or sensors to detect and mitigate fault injection attacks through voltage or clock manipulation. Modern devices implement security features like secure boot that establish a chain of trust by verifying cryptographic signatures before executing firmware. These mechanisms rely on the assumption that decision points in security-critical code paths will execute correctly.

Voltage glitching involves briefly manipulating the power supply to cause the processor to malfunction in a controlled way, while clock glitching introduces anomalies in the clock signal timing. Both techniques can cause the CPU to skip instructions, corrupt computations, or misread values at critical decision points. Without dedicated detection circuitry, the hardware has no way to recognize that it is under attack.

The fundamental issue is a design trade-off: adding glitch detection circuits increases silicon area, power consumption, and design complexity. However, without these protections, attackers with physical access can bypass even well-implemented software security controls by attacking the hardware execution environment itself.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read protected memory and extract confidential data such as cryptographic keys or secure boot verification values |
| **Integrity** | The attacker can bypass signature verification, modify memory, and execute unauthorized code |
| **Availability** | The attacker can cause unpredictable system behavior or denial of service through repeated fault injection |

## Attack Scenario

An attacker with physical access to a device targets the secure boot process. The attacker connects voltage or clock glitching equipment to the device's power supply or clock lines. During the boot sequence, the bootloader performs a cryptographic signature verification before loading the runtime firmware. The attacker precisely times a voltage glitch or clock anomaly to occur at the exact moment when the CPU evaluates the signature comparison result. Due to the absence of glitch detection circuitry, the processor's internal state becomes corrupted, causing the conditional branch to take the wrong path and load unsigned or malicious firmware despite the signature mismatch. After bypassing secure boot, the attacker gains access to protected system assets, can read secure eFuses, or install persistent malware.

## MITRE Reference Data

### Description

The device does not contain or contains incorrectly implemented circuitry or sensors to detect and mitigate voltage and clock glitches and protect sensitive information or software contained on the device.

### Extended Description

A device might support features such as secure boot which are supplemented with hardware and firmware support. This involves establishing a chain of trust, starting with an immutable root of trust by checking the signature of the next stage (culminating with the OS and runtime software) against a golden value before transferring control. The intermediate stages typically set up the system in a secure state by configuring several access control settings. Similarly, security logic for exercising a debug or testing interface may be implemented in hardware, firmware, or both. A device needs to guard against fault attacks such as voltage glitches and clock glitches that an attacker may employ in an attempt to compromise the system.

If the signatures match, the bootloader loads runtime firmware. If there is no match, an error halt occurs.
