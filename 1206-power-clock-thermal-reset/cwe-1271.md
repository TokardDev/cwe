# CWE-1271 - Uninitialized Value on Reset for Registers Holding Security Settings
## CWE Explanation

This weakness occurs when hardware registers responsible for critical security settings, such as lock bits or access control configurations, do not have a defined reset value. When a device is powered on or reset, these registers may contain indeterminate data or retain values from a previous state (remanence) rather than defaulting to a secure "closed" or "locked" state.

In digital logic design, flip-flops and registers require an explicit reset signal (synchronous or asynchronous) to force them into a known state. If this is omitted for security-critical logic, a temporal gap is created between the release of the reset signal and the moment firmware or other logic explicitly initializes these registers. During this window, the device operates based on the random or residual values in these registers.

This vulnerability is particularly dangerous because it allows attackers to bypass security mechanisms by simply resetting the device. If the random power-up state happens to correspond to an "unlocked" or "privileged" state, the attacker can gain unauthorized access before the system has a chance to apply its security policy. This is often exploited through "power glitching" or repeated reset cycling to statistically achieve the desired insecure state.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can access sensitive internal state or debug interfaces if the lock bit powers up in an unlocked state. |
| **Integrity** | The attacker can modify system configuration or firmware if write-protection registers are not properly initialized. |
| **Availability** | Low impact (*System instability is possible if configuration registers power up with invalid settings*). |

## Attack Scenario

An attacker is able to physically control the power or reset line of a target device that uses a hardware lock bit to protect its JTAG debug interface. In the next step, the attacker repeatedly power-cycles or resets the device while attempting to communicate with the debug interface. The specific flip-flop controlling the "lock" signal does not have an asynchronous reset connection, meaning its state upon power-up is determined by thermal noise and process variations (effectively random). By continuously resetting the device, the attacker eventually encounters a case where the flip-flop settles into the "unlocked" state (logic 0) instead of the secure "locked" state (logic 1). During the brief window before the boot code runs and explicitly sets the lock bit, the attacker successfully establishes a JTAG connection, bypassing the security mechanism and gaining full control over the processor.

## MITRE Reference Data

### Description

Security-critical logic is not set to a known value on reset.

### Extended Description

When the device is first brought out of reset, the state of registers will be indeterminate if they have not been initialized by the logic. Before the registers are initialized, there will be a window during which the device is in an insecure state and may be vulnerable to attack.

### Demonstrative Examples

Example 1

Shown below is a positive clock edge triggered flip-flop used to implement a lock bit for test and debug interface. When the circuit is first brought out of reset, the state of the flip-flop will be unknown until the enable input and D-input signals update the flip-flop state. In this example, an attacker can reset the device until the test and debug interface is unlocked and access the test interface until the lock signal is driven to a known state by the logic.

```verilog
always @(posedge clk) begin
  if (en) lock_jtag <= d;
end
```

The flip-flop can be set to a known value (0 or 1) on reset, but requires that the logic explicitly update the output of the flip-flop if the reset signal is active.

```verilog
always @(posedge clk) begin
  if (~reset) lock_jtag <= 0;
  else if (en) lock_jtag <= d;
end
```

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Implementation | - | Design checks should be performed to identify any uninitialized flip-flops used for security-critical functions. | - |
| Architecture and Design | - | All registers holding security-critical information should be set to a specific value on reset. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Include a hardware reset-verification mechanism that confirms all security-relevant registers have reached their expected secure default values before allowing boot to proceed | Critical |
| Verification | Establish an automated design-rule check in the build flow that ensures every register in a security domain has a defined and documented reset value | Critical |
| Architecture and Design | Design the reset value of debug interface controls (e.g., JTAG lock) to the most restrictive state (locked), requiring explicit authenticated unlock by trusted firmware. | High |
| Testing | Perform reset-state verification testing across all reset types (POR, warm reset, watchdog reset) to confirm that all security registers initialize to their defined secure values. | High |
| Implementation | Where possible, use power-on-reset (POR) circuits with deterministic initialization rather than relying on register remanence or indeterminate states. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
