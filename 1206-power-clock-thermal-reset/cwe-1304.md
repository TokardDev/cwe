# CWE-1304 - Improperly Preserved Integrity of Hardware Configuration State During a Power Save/Restore Operation

## CWE Explanation

This weakness arises when a hardware component or Intellectual Property (IP) saves its configuration state to persistent storage (like flash memory or always-on RAM) during a power-save operation but fails to verify the integrity of that state upon restoration. To optimize wake-up times, systems often dump their internal register state to external memory before entering sleep modes and reload it when powering back up.

If the storage medium used for this state preservation is accessible to an attacker (physically or via other interfaces), the saved state can be tampered with while the system is dormant. Without an integrity check (such as a cryptographic hash or digital signature) during the restore process, the hardware will blindly accept and apply the modified configuration.

The root cause is the implicit trust placed in the persistent storage medium during the power cycle. This allows an attacker to bypass runtime security controls by modifying the system's state "offline," effectively injecting a malicious configuration that the hardware adopts as its own valid state upon waking.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can gain privileges or assume identity |
| **Integrity** | The attacker can bypass protection mechanisms and alter execution logic |
| **Availability** | The attacker can cause instability, crashes, or permanent damage to the hardware |

## Attack Scenario

An attacker is able to access the persistent storage where a hardware block saves its context during sleep modes. In the next step, the system initiates a power-save operation, writing its current security configuration and internal state to this external memory. While the system is in the low-power state, the attacker modifies the stored configuration data, disabling specific security bits or elevating privilege levels defined in the saved register map. When the system wakes up and performs the restore operation, it reads the corrupted state data back into its internal registers without validating its integrity. Consequently, the hardware resumes operation in a compromised state defined by the attacker, effectively bypassing boot-time security checks and runtime protections.

## MITRE Reference Data

### Description

The product performs a power save/restore operation, but it does not ensure that the integrity of the configuration state is maintained and/or verified between the beginning and ending of the operation.

### Extended Description

Before powering down, the Intellectual Property (IP) saves current state (S) to persistent storage such as flash or always-on memory in order to optimize the restore operation.  During this process, an attacker with access to the persistent storage may alter (S) to a configuration that could potentially modify privileges, disable protections, and/or cause damage to the hardware. If the IP does not validate the configuration state stored in persistent memory, upon regaining power or becoming operational again, the IP could be compromised through the activation of an unwanted/harmful configuration.

### Demonstrative Examples

Example 1

The following pseudo code demonstrates the power save/restore workflow which may lead to weakness through a lack of validation of the config state after restore.

```c
// (bad code)
void save_config_state()
{
	void* cfg;

	cfg = get_config_state();
	save_config_state(cfg);

	go_to_sleep();
}

void restore_config_state()
{
	void* cfg;
	cfg = get_config_file();
	load_config_file(cfg);
}
```

The following pseudo-code is the proper workflow for the integrity checking mitigation:

```c
// (good code)
void save_config_state()
{
	void* cfg;
	void* sha;

	cfg = get_config_state();
	save_config_state(cfg);

	// save hash(cfg) to trusted location
	sha = get_hash_of_config_state(cfg);
	save_hash(sha); 

	go_to_sleep();
}

void restore_config_state()
{
	void* cfg;
	void* sha_1, sha_2;

	cfg = get_config_file();
	// restore hash of config from trusted memory
	sha_1 = get_persisted_sha_value();

	sha_2 = get_hash_of_config_state(cfg);
	if (sha_1 != sha_2)
	{
		assert_error_and_halt();
	}

	load_config_file(cfg);
}
```

It must be noted that in the previous example of good pseudo code, the memory (where the hash of the config state is stored) must be trustworthy while the hardware is between the power save and restore states.
