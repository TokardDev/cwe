## CWE Explanation

This weakness arises when software interfaces allow unrestricted access to hardware features like power management, clock control, or memory access patterns. While these features are necessary for functionality such as energy efficiency and performance tuning, they can be abused to induce physical phenomena that compromise security.

Typically, physical attacks like fault injection or side-channel analysis require physical access to the device. However, this weakness effectively exposes a "software knob" that duplicates the effects of physical access. For example, software might rapidly toggle memory rows to induce bit flips (Rowhammer) or undervolt the processor to cause computation errors (Plundervolt/CLKSCREW).

The root cause is a failure to strictly isolate or limit the range of these hardware features when exposed to untrusted software. When the hardware design blindly fulfills software requests without verifying safety margins or privilege levels, it bridges the gap between the logical and physical domains, allowing software to violate hardware integrity.

| **Confidentiality** | The attacker can extract cryptographic keys or sensitive data via side-channel analysis enabled by power metering interfaces. |

| **Availability** | The attacker can cause system instability, crashes, or potentially permanent physical damage to the hardware. |

## Attack Scenarios

### Scenario 1
An attacker is able to execute code that performs rapid, repeated memory access to specific physical addresses. In the next step, the attacker enters a tight loop that repeatedly activates the same row in dynamic random-access memory (DRAM). Due to the physical proximity of memory cells and the absence of targeted refresh mitigations, this rapid activation causes electromagnetic coupling effects that leak charge into adjacent rows. This results in bit flips in the victim's memory space (Rowhammer), potentially corrupting data or altering page table entries to gain privilege escalation.

### Scenario 2
An attacker is able to access software interfaces that control the device's operating voltage and clock frequency. In the next step, the attacker commands the hardware to reduce the voltage or increase the clock frequency beyond stable operating margins while a victim process (e.g., a trusted execution environment) performs a critical operation like a cryptographic signature check. This induces transient timing or voltage faults in the processor logic. This results in the processor misinterpreting instructions or data, allowing the attacker to bypass authentication checks or recover cryptographic keys.

### Scenario 3
An attacker is able to access high-resolution hardware power metering interfaces (e.g., RAPL) from an unprivileged context. In the next step, the attacker triggers a cryptographic operation in a victim process and simultaneously samples the power consumption data provided by the hardware interface. By analyzing the correlation between the power fluctuations and the cryptographic operations, the attacker performs a software-based differential power analysis (DPA). This results in the extraction of the secret key used by the victim, effectively performing a side-channel attack without physical probes.

## MITRE Reference Data

### Description

The product provides software-controllable device functionality for capabilities such as power and clock management, but it does not properly limit functionality that can lead to modification of hardware memory or register bits, or the ability to observe physical side channels.

### Extended Description

It is frequently assumed that physical attacks such as fault injection and side-channel analysis require an attacker to have physical access to the target device. This assumption may be false if the device has improperly secured power management features, or similar features. For mobile devices, minimizing power consumption is critical, but these devices run a wide variety of applications with different performance requirements. Software-controllable mechanisms to dynamically scale device voltage and frequency and monitor power consumption are common features in today's chipsets, but they also enable attackers to mount fault injection and side-channel attacks without having physical access to the device.

Fault injection attacks involve strategic manipulation of bits in a device to achieve a desired effect such as skipping an authentication step, elevating privileges, or altering the output of a cryptographic operation. Manipulation of the device clock and voltage supply is a well-known technique to inject faults and is cheap to implement with physical device access. Poorly protected power management features allow these attacks to be performed from software. Other features, such as the ability to write repeatedly to DRAM at a rapid rate from unprivileged software, can result in bit flips in other memory locations (Rowhammer, [REF-1083]).

Side channel analysis requires gathering measurement traces of physical quantities such as power consumption. Modern processors often include power metering capabilities in the hardware itself (e.g., Intel RAPL) which if not adequately protected enable attackers to gather measurements necessary for performing side-channel attacks from software.

### Demonstrative Examples

<p class="subheading">Example 1</p>

This example considers the Rowhammer problem [REF-1083]. The Rowhammer issue was caused by a program in a tight loop writing repeatedly to a location to which the program was allowed to write but causing an adjacent memory location value to change.

```other
(bad code)
Continuously writing the same value to the same address causes the value of an adjacent location to change value.
```

Preventing the loop required to defeat the Rowhammer exploit is not always possible:

```other
(good code)
Redesign the RAM devices to reduce inter capacitive coupling making the Rowhammer exploit impossible.
```

While the redesign may be possible for new devices, a redesign is not possible in existing devices. There is also the possibility that reducing capacitance with a relayout would impact the density of the device resulting in a less capable, more costly device.
