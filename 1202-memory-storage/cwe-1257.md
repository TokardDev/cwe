# CWE-1257 - Improper Access Control Applied to Mirrored or Aliased Memory Regions

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read protected memory contents by accessing aliased addresses that bypass access control enforcement |
| **Integrity** | The attacker can modify protected memory contents by writing to aliased addresses that bypass access control enforcement |
| **Availability** | System instability is possible due to unauthorized memory modifications affecting critical system state |

## Attack Scenarios

### Scenario 1
An attacker is able to interact with a hardware system where memory regions are aliased due to address decoder logic masking higher-order address bits. In the next step, the attacker discovers that while the primary memory address range (e.g., 0x0000 - 0x0FFF) is protected by access control policies blocking unprivileged access, the aliased address ranges (e.g., 0x1000 - 0x1FFF, 0x2000 - 0x2FFF) mapping to the same physical memory cells are not subject to the same protection. The attacker reads or writes to protected registers through the unprotected aliased addresses, bypassing the memory protection unit and gaining unauthorized access to sensitive configuration or data.

### Scenario 2
An attacker is able to modify memory region size configuration or address mapping registers in a System-on-Chip design. In the next step, the attacker creates new memory aliases by reconfiguring the address decoder to map additional system addresses to an already-protected memory region. Since the memory protection unit policies are not automatically synchronized to cover newly created aliases, the attacker accesses the protected memory through the maliciously created alias, extracting confidential data or modifying critical system parameters.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation | - | The checks should be applied for consistency access rights between primary memory regions and any mirrored or aliased memory regions. If different memory protection units (MPU) are protecting the aliased regions, their protected range definitions and policies should be synchronized. | - |
| Architecture and Design; Implementation | - | The controls that allow enabling memory aliases or changing the size of mapped memory regions should only be programmable by trusted software components. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement a unified access control enforcement point in the address decoder that applies protection policies based on the resolved physical address, not the requested virtual/aliased address | Critical |
| Architecture and Design | Restrict memory alias configuration and address mapping registers to trusted firmware only, using hardware-enforced privilege checks and lock bits | Critical |
| Implementation | Synchronize MPU policies across all aliased regions automatically in hardware whenever a protection policy is updated on the primary region | High |
| Testing | Perform exhaustive address-space sweeps in verification to detect unprotected aliases, including testing with all valid address bit combinations that could resolve to protected physical memory | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
