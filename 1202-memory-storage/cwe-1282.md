# CWE-1282 - Assumed-Immutable Data is Stored in Writable Memory


## Related CVEs (from NVD)

| CVE | Description |
|-----|-------------|
| [CVE-2019-25358](https://www.cve.org/CVERecord?id=CVE-2019-25358) | FileOptimizer DoS via assumed-immutable data stored in writable memory |

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Low impact (*Confidentiality loss may occur if modified code is used to exfiltrate data*) |
| **Integrity** | The attacker can modify trusted assets such as bootloaders, cryptographic keys, and configuration data |
| **Availability** | Low impact (*System may fail to boot if critical data is corrupted*) |

## Attack Scenarios

### Scenario 1
An attacker with physical or privileged access to a device identifies that the first-stage bootloader is stored in writable flash memory instead of ROM or OTP memory. The attacker reprograms the flash memory to inject a malicious bootloader that bypasses secure boot verification. Because the system treats this memory region as containing immutable trusted code, no integrity check is performed on the bootloader itself. The attacker gains persistent root-of-trust compromise, allowing execution of arbitrary code before any security mechanisms are initialized.

### Scenario 2
An attacker targets a device that stores golden hash digests in writable EEPROM to verify firmware integrity during boot. The attacker exploits a vulnerability to gain write access to the EEPROM region containing these digests. By replacing the legitimate digest with one computed from malicious firmware, the attacker ensures that the compromised firmware passes all integrity verification checks. The device subsequently boots the attacker-controlled firmware, believing it to be authentic and unmodified.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Implementation | - | All immutable code or data should be programmed into ROM or write-once memory. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Store root-of-trust code (first-stage bootloader) and golden hash digests in ROM or one-time-programmable (OTP) memory that cannot be modified after provisioning | Critical |
| Architecture and Design | If writable storage must be used for certain configuration data, protect it with hardware write-lock mechanisms that can only be released through authenticated privileged operations | High |
| Implementation | Implement a hardware-rooted chain of trust where each stage cryptographically verifies the next, using keys stored in OTP/ROM, so that modification of any single writable asset is detectable | Critical |
| Testing | Validate in post-silicon testing that all memory regions designated as immutable are truly non-writable under all access paths, including debug, DMA, and alternate bus master scenarios | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
