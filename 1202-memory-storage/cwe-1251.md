# CWE-1251 - Mirrored Regions with Different Values

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read unauthorized memory |
| **Integrity** | The attacker can modify protected memory |
| **Availability** | Low impact (*System instability is possible due to inconsistent states*) |

## Attack Scenarios

### Scenario 1
An attacker is able to interact with a shadow copy of a file on a multi-core processor system where a shadow register file is implemented for performance. A trusted process modifies the values in the original copy (e.g., revoking). However, due to an internal logic failure or state error, the original copy fails to trigger or send the necessary "update" request to its shadow copies. Consequently, the shadow regions are never notified of the change and indefinitely retain the stale and permissive values. The attacker accesses the shadow copy, exploiting the obsolete state to bypass the security controls that were intended to be set by the update.

### Scenario 2
An attacker is able to interact with a shadow copy of a file on a multi-core processor system where a shadow register file is implemented for performance. A trusted user or process updates a primary resource and the system correctly initiates the synchronization process. The original copy dutifully transmits the "update" request to the mirrored regions. However, due to a fault on the receiving end, the shadow copy fails to process or execute this request. As a result, the primary copy reflects the secure state, while the shadow copy remains out of sync. The attacker accesses the shadow copy, exploiting the obsolete state to bypass the security controls that were intended to be set by the update.

### Scenario 3
An attacker is able to interact with a shadow copy of a file on a multi-core processor system where a shadow register file is implemented for performance. A trusted user or process updates a primary resource and the system correctly initiates the synchronization process and is designed to execute this update faithfully. However, a physical or logical latency exists during the time period between the primary update and the shadow update. An attacker triggers a race condition by initiating a request to the shadow copy specifically during this window. If the attacker's request reaches the target before the update process concludes, the attacker successfully exploits the "old" values, winning the race against the synchronization mechanism and abusing the obsolete state to bypass the security controls.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Whenever there are multiple, physically different copies of the same value that might change and the process to update them is not instantaneous and atomic, it is impossible to assert that the original and shadow copies will always be in sync - there will always be a time period when they are out of sync. To mitigate the consequential risk, the recommendations essentially are: Make this out-of-sync time period as small as possible, and make the update process as robust as possible. | Moderate |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement atomic or near-atomic synchronization mechanisms (e.g., hardware handshake protocols) between primary and shadow registers to minimize the window of inconsistency | Critical |
| Architecture and Design | Block access to shadow copies during synchronization by using hardware lock signals that gate reads until the update is confirmed complete | High |
| Implementation | Add integrity checks (e.g., version counters or checksums) that allow consumers of mirrored data to detect stale values and retry or fail safely | High |
| Testing | Include race-condition-focused verification scenarios in pre-silicon testing that exercise concurrent access to primary and shadow copies during updates | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
