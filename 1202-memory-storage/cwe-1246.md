# CWE-1246 - Improper Write Handling in Limited-write Non-Volatile Memories

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Low impact |
| **Integrity** | The attacker already have a write permission on the memory |
| **Availability** | The attacker can cause permanent hardware failure (storage may become unusable) |

## Attack Scenario

An attacker is able to execute a program with write access on limited-write non-volatile memory (e.g. NAND Flash or EEPROM). In the next step, the attacker initiates a high frequency loop of write and erase operations targeting a specific address or a small set of addresses. Due to the absence or incorrect implementation of wear leveling algorithms (or the use of an algorithm that the attacker can bypass), the memory controller directs these operations repeatedly to the same physical memory block rather than distributing the wear across the device. Consequently, the targeted physical memory cells exceed their maximum program/erase cycles and fail prematurely. This results in permanent physical damage to the memory, leading the storage to become unreliable within a much shorter time than would normally be expected, and potentially rendering the device permanently inoperable (persistent DoS).

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Implementation; Testing | - | Include secure wear leveling algorithms and ensure they may not be bypassed. | High |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement hardware-enforced wear leveling in the memory controller that distributes write/erase operations across all physical blocks, preventing software from targeting specific cells | Critical |
| Architecture and Design | Add write rate limiting or throttling mechanisms in the memory controller to detect and block abnormally high-frequency write/erase patterns to the same address range | High |
| Implementation | Implement bad-block management with health monitoring that tracks per-block erase counts and triggers alerts or protective actions when thresholds are approached | High |
| Testing | Validate wear leveling under adversarial conditions in pre-silicon and post-silicon testing, including fuzzing write patterns to confirm no bypass paths exist | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
