# CWE-1266 - Improper Scrubbing of Sensitive Data from Decommissioned Device

## CWE Explanation

This weakness occurs when a device lacks a reliable mechanism to securely erase or "scrub" sensitive data before it is taken out of service (decommissioned). Hardware devices often store sensitive information such as user credentials, network keys, configuration secrets, or proprietary manufacturer data in non-volatile memory (e.g., Flash, EEPROM).

If the device does not provide a built-in function to overwrite this data, or if the procedure is not documented or accessible to the administrator, the data remains on the physical medium after the device is discarded. This allows an attacker who acquires the physical device (e.g., from e-waste, second-hand markets) to extract the data directly from the memory chips, potentially compromising the previous owner's network or exposing proprietary intellectual property.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | High impact (*The attacker can read sensitive data from the physical memory*) |
| **Integrity** | Low impact (*Integrity of the data on the decommissioned device is not the primary concern*) |
| **Availability** | Low impact (*Availability of the decommissioned device is not relevant*) |

## Attack Scenario

An attacker acquires a decommissioned hardware device from an electronic waste recycling facility or an online marketplace. The device was previously used in a secure environment and contains sensitive network credentials and configuration keys in its non-volatile memory. The previous administrator attempted to reset the device before disposal but found no "secure erase" or "factory reset" function that cleared the internal EEPROM, or the function was undocumented. In the next step, the attacker desolders the EEPROM chip or connects via a JTAG interface to dump the memory contents. Because the sensitive data was not scrubbed, the attacker successfully extracts the credentials and uses them to compromise the organization's active network.

## MITRE Reference Data

### Description

The product does not properly provide a capability for the product administrator to remove sensitive data at the time the product is decommissioned. A scrubbing capability could be missing, insufficient, or incorrect.

### Extended Description

When a product is decommissioned - i.e., taken out of service - best practices or regulatory requirements may require the administrator to remove or overwrite sensitive data first, i.e. "scrubbing." Improper scrubbing of sensitive data from a decommissioned device leaves that data vulnerable to acquisition by a malicious actor. Sensitive data may include, but is not limited to, device/manufacturer proprietary information, user/device credentials, network configurations, and other forms of sensitive data.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | — | Functionality to completely scrub data from a product at the conclusion of its lifecycle should be part of the design phase. Trying to add this function on top of an existing architecture could lead to incomplete removal of sensitive information/data. | — |
| Policy | — | The manufacturer should describe the location(s) where sensitive data is stored and the policies and procedures for its removal. This information may be conveyed, for example, in an Administrators Guide or a Statement of Volatility. | — |
| Implementation | — | If the capability to wipe sensitive data isn't built-in, the manufacturer may need to provide a utility to scrub sensitive data from storage if that data is located in a place which is non-accessible by the administrator. One example of this could be when sensitive data is stored on an EEPROM for which there is no user/admin interface provided by the system. | — |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design a hardware-assisted secure erase function that performs a cryptographic erase (destroying the encryption key) or multi-pass overwrite of all non-volatile memory regions (Flash, EEPROM, OTP where possible), triggered by a dedicated physical or authenticated command. | Critical |
| Implementation | Ensure that the secure erase routine covers all locations where sensitive data may reside, including wear-leveled flash pages, hidden OTP areas, and any mirrored or cached copies in SRAM that persist across resets. | Critical |
| Policy | Publish a Statement of Volatility (SoV) document that enumerates every non-volatile storage element on the device, the type of data stored, and the exact procedure to scrub each element. | High |
| Testing | Validate the secure erase function by performing a full memory dump after the erase procedure and verifying that no residual sensitive data patterns remain. Include this as a mandatory test in the product qualification plan. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
