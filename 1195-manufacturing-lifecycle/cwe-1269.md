## CWE Explanation

This weakness occurs when a hardware product is shipped to end-users while still configured in a "pre-production" or "manufacturing" state. During the manufacturing process, devices often have special modes enabled that allow for deep testing, debugging, and configuration. These modes provide elevated privileges, such as the ability to read/write arbitrary memory, bypass cryptographic signature checks, or access internal registers that are normally locked.

The transition from manufacturing to production usually involves a permanent, irreversible action, such as blowing a specific fuse (e.g., "Manufacturing Complete" fuse) or setting a lock bit in One-Time Programmable (OTP) memory. If this step is missed due to a process failure or operator error, the device remains in its vulnerable manufacturing configuration. Attackers can then exploit these powerful interfaces to extract sensitive secrets (like firmware encryption keys), modify the firmware, or completely bypass the device's security model.

## Attack Scenario

An attacker obtains a consumer device that was accidentally shipped in "Manufacturing Mode." This mode is typically used by the factory to perform initial provisioning and testing. Because the "Manufacturing Complete" fuse was not blown during the final stage of assembly, the device's JTAG interface and a special UART diagnostic console remain active and unlocked. In the next step, the attacker connects to these interfaces and discovers they have full read/write access to the device's memory and registers, bypassing the secure boot mechanism. The attacker uses this access to extract the global encryption key used for firmware updates, allowing them to create and install malicious firmware on this and other similar devices.

## MITRE Reference Data

### Description

The product released to market is released in pre-production or manufacturing configuration.

### Extended Description

Products in the pre-production or manufacturing stages are configured to have many debug hooks and debug capabilities, including but not limited to:

* Ability to override/bypass various cryptographic checks (including authentication, authorization, and integrity)
* Ability to read/write/modify/dump internal state (including registers and memory)
* Ability to change system configurations
* Ability to run hidden or private commands that are not allowed during production (as they expose IP).

The above is by no means an exhaustive list, but it alludes to the greater capability and the greater state of vulnerability of a product during it's preproduction or manufacturing state.

Complexity increases when multiple parties are involved in executing the tests before the final production version. For example, a chipmaker might fabricate a chip and run its own preproduction tests, following which the chip would be delivered to the Original Equipment Manufacturer (OEM), who would now run a second set of different preproduction tests on the same chip. Only after both of these sets of activities are complete, can the overall manufacturing phase be called "complete" and have the "Manufacturing Complete" fuse blown. However, if the OEM forgets to blow the Manufacturing Complete fuse, then the system remains in the manufacturing stage, rendering the system both exposed and vulnerable.

### Demonstrative Examples

#### Example 1

This example shows what happens when a preproduction system is made available for production.

```other
(bad code)
Suppose the chipmaker has a way of scanning all the internal memory (containing chipmaker-level secrets) during the manufacturing phase, and the way the chipmaker or the Original Equipment Manufacturer (OEM) marks the end of the manufacturing phase is by blowing a Manufacturing Complete fuse. Now, suppose that whoever blows the Manufacturing Complete fuse inadvertently forgets to execute the step to blow the fuse.
```

An attacker will now be able to scan all the internal memory (containing chipmaker-level secrets).

```other

Blow the Manufacturing Complete fuse.
```

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Manufacturing | Implement an automated end-of-line test station that verifies the Manufacturing Complete fuse is blown before a device passes final quality gate. Devices that fail this check must be quarantined and cannot be shipped. | Critical |
| Architecture and Design | Design the SoC so that when the Manufacturing Complete fuse is not blown, the device boots into a visibly restricted mode (e.g., displays a warning, limits network connectivity) making it immediately obvious that it is not production-ready. | Critical |
| Implementation | Ensure that blowing the Manufacturing Complete fuse irreversibly disables all manufacturing/debug interfaces (JTAG, UART diagnostic console, scan chains) and that no software-only mechanism can re-enable them. | High |
| Testing | Include a post-manufacturing audit step where a random sample of shipped devices is tested to confirm the Manufacturing Complete fuse state, providing statistical assurance that the production process is functioning correctly. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
