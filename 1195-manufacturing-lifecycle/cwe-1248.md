## CWE Explanation

This weakness occurs when semiconductor devices that implement security-critical logic contain physical defects that cause incorrect operation. These defects can originate during manufacturing or develop over time due to operational stress, and they manifest as stuck-at faults where signals or registers are permanently fixed to either 0 or 1 instead of switching normally.

The root causes of semiconductor defects are diverse and span the entire device lifecycle. Manufacturing-related defects include encapsulation failures, die-attach problems, wire-bond failures, and bulk-silicon imperfections. Operational factors such as electromigration, aluminum corrosion, oxide-layer degradation, and thermal or electrical stress can cause defects to emerge after deployment, particularly when devices operate outside their specified temperature, voltage, or humidity ranges.

When these defects occur in hardware modules responsible for security functions, they can fundamentally compromise the security guarantees the hardware was designed to provide. For example, a stuck-at-1 fault in an access control filter could permanently grant access regardless of the intended policy, effectively bypassing security mechanisms at the hardware level. Unlike software vulnerabilities that can be patched, hardware defects of this nature typically cannot be remediated after deployment.

## Attack Scenario

An attacker targets a system-on-chip where a network-on-chip firewall implements access control to peripherals from IP cores capable of mastering transactions. A manufacturing defect in the firewall's comparison logic causes a stuck-at fault that permanently sets the access decision output to "allow". As a result, access control checks always pass regardless of the requesting core's actual permissions. The attacker, operating from a low-privilege IP core, issues transactions to protected peripherals that should be denied. Due to the hardware defect, the firewall grants access, allowing the attacker to read or write protected memory regions, extract cryptographic keys, or manipulate security-sensitive configuration registers.

## MITRE Reference Data

### Description

The security-sensitive hardware module contains semiconductor defects.

### Extended Description

A semiconductor device can fail for various reasons. While some are manufacturing and packaging defects, the rest are due to prolonged use or usage under extreme conditions. Some mechanisms that lead to semiconductor defects include encapsulation failure, die-attach failure, wire-bond failure, bulk-silicon defects, oxide-layer faults, aluminum-metal faults (including electromigration, corrosion of aluminum, etc.), and thermal/electrical stress. These defects manifest as faults on chip-internal signals or registers, have the effect of inputs, outputs, or intermediate signals being always 0 or always 1, and do not switch as expected.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Testing | - | While semiconductor-manufacturing companies implement several mechanisms to continuously improve the semiconductor manufacturing process to ensure reduction of defects, some defects can only be fixed after manufacturing. Post-manufacturing testing of silicon die is critical. Fault models such as stuck-at-0 or stuck-at-1 must be used to develop post-manufacturing test cases and achieve good coverage. Once the silicon packaging is done, extensive post-silicon testing must be performed to ensure that hardware logic implementing security functionalities is defect-free. | - |
| Operation | - | Operating the hardware outside device specification, such as at extremely high temperatures, voltage, etc., accelerates semiconductor degradation and results in defects. When these defects manifest as faults in security-critical, hardware modules, it results in compromise of security guarantees. Thus, operating the device within the specification is important. | - |

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Testing | Develop dedicated post-silicon test vectors targeting stuck-at faults in security-critical modules (access control filters, key registers, boot ROM comparators) and require 100% fault coverage on these paths before shipment. | Critical |
| Architecture and Design | Implement hardware redundancy (e.g., dual-rail logic or Triple Modular Redundancy) for security-critical decision paths such as access control checks and cryptographic verification, so that a single stuck-at fault cannot silently bypass security. | High |
| Operation | Define and enforce operational envelopes (temperature, voltage, clock frequency) with on-chip sensors that trigger a safe shutdown or alert when thresholds are exceeded, preventing accelerated degradation of security logic. | High |
| Testing | Integrate Built-In Self-Test (BIST) routines for security-critical logic that can be executed periodically in the field to detect latent defects that develop after deployment. | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
