## Attack Scenarios

### Scenario 1
An attacker is able to corrupt a low trust application and attempts to read a memory address restricted to the kernel. In this scenario, the processor performs the access control check (verifying if the address is mapped to user or kernel space) in parallel with the memory read operation. Consequently, the processor transiently returns the restricted kernel data from the shared cache before the permission check officially fails. The attacker utilizes a gadget to transmit this result over a covert channel before the pipeline is flushed, bypassing the memory isolation.

### Scenario 2
An attacker is able to execute unprivileged code and attempts to read a system register restricted to a higher privilege level. In a normal case, the hardware prevents this access and would raise an exception. However, the attacker manipulates the branch prediction system (e.g. via a conditional branch CBZ [Compare and Branch on Zero instruction]) to mispredict a path where the read instruction is executed transiently. During this transient window, the processor reads the value of the restricted register before the misprediction is resolved. The attacker then uses a subsequent memory access instruction to encode this restricted value into a cache covert channel, making it retrievable even after the pipeline is flushed.

### Scenario 3
An attacker runs on a hardware thread that shares physical resources with a victim thread on the same processor core. The victim's data passes through shared microarchitectural structures known as fill buffers (used for temporary storage between caches and memory). The attacker triggers a transient execution event, specifically via a faulting operation or a microcode assist. This event allows the attacker to transiently read data currently residing in the shared fill buffer, effectively bypassing the logical isolation between the two threads. The attacker extracts this data using a side-channel analysis before the processor clears the buffer state.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Hardware designers may choose to engineer the processor's pipeline to prevent architecturally restricted data from being used by operations that can execute transiently. | High |
| Architecture and Design | - | Hardware designers may choose not to share microarchitectural resources that can contain sensitive data, such as fill buffers and store buffers. | Moderate |
| Architecture and Design | - | Hardware designers may choose to sanitize specific microarchitectural state (for example, store buffers) when the processor transitions to a different context. Alternatively, the hardware may expose instruction(s) that allow software to sanitize microarchitectural state according to the user or system administrator's threat model. | Moderate |
| Architecture and Design | - | The hardware designer can attempt to prevent transient execution from causing observable discrepancies in specific covert channels. | Limited |
| Architecture and Design | - | Software architects may design software to enforce strong isolation between different contexts. For example, kernel page table isolation (KPTI) mitigates the Meltdown vulnerability by separating user-mode page tables from kernel-mode page tables. | Limited |
| Build and Compilation | - | If the weakness is exposed by a single instruction (or a small set of instructions), then the compiler (or JIT, etc.) can be configured to prevent the affected instruction(s) from being generated. | Limited |
| Build and Compilation | - | Use software techniques (including the use of serialization instructions) that are intended to reduce the number of instructions that can be executed transiently after a processor event or misprediction. | Incidental |
| Implementation | - | System software can mitigate this weakness by invoking state-sanitizing operations when switching from one context to another, according to the hardware vendor's recommendations. | Limited |
| System Configuration | - | Some systems may allow the user to disable (for example, in the BIOS) sharing of the affected resource. | Limited |
| System Configuration | - | Some systems may allow the user to disable (for example, in the BIOS) microarchitectural features that allow transient access to architecturally restricted data. | Limited |
| Patching and Maintenance | - | The hardware vendor may provide a patch to sanitize the affected shared microarchitectural state when the processor transitions to a different context. | Moderate |
| Patching and Maintenance | - | This kind of patch may not be feasible or implementable for all processors or all weaknesses. | Limited |
| Requirements | - | Processor designers, system software vendors, or other agents may choose to restrict the ability of unprivileged software to access to high-resolution timers that are commonly used to monitor covert channels. | Defense in Depth |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Engineer the processor pipeline to ensure that architecturally restricted data (kernel memory, enclave data) cannot be forwarded to transiently executing instructions, even during the window between access and permission check completion | Critical |
| Architecture and Design | Partition shared microarchitectural structures (L1D cache, fill buffers, store buffers, load ports) between SMT threads and across privilege boundaries to prevent cross-domain data leakage | Critical |
| Architecture and Design | Implement automatic hardware-level sanitization of shared microarchitectural buffers on context switches and privilege transitions (syscall, interrupt, VM exit) | High |
| Implementation | Deploy kernel page table isolation (KPTI) and ensure system software invokes appropriate state-sanitizing instructions (e.g., VERW for MDS mitigations) on every context switch | High |
| System Configuration | Provide BIOS/firmware options to disable SMT on security-critical deployments where cross-thread data leakage risk outweighs the performance benefit | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
