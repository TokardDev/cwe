# CWE-1422 - Exposure of Sensitive Information caused by Incorrect Data Forwarding during Transient Execution


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2020-0551](https://www.cve.org/CVERecord?id=CVE-2020-0551) | A fault, microcode assist, or abort may allow transient
				load operations to forward malicious stale data to dependent
				operations executed by a victim, causing the victim to unintentionally
				access and potentially expose its own data over a covert channel. |
| [CVE-2020-8698](https://www.cve.org/CVERecord?id=CVE-2020-8698) | A fast store forwarding predictor may allow store
				operations to forward incorrect data to transient load operations,
				potentially exposing data over a covert channel. |

## Attack Scenarios

### Scenario 1
An attacker is able to identify a vulnerable code sequence (gadget) within a victim process or a high-trust application. In the next step, the attacker triggers a processor event, such as a Page Fault, while the processor is handling a data store operation. Due to the specific hardware design, the processor incorrectly forwards data from an internal buffer (like the store buffer) to a subsequent transient operation before the fault is resolved. This forwarded data is often stale or incorrect, but the transient instruction uses it to perform a memory access dependent on this value. Consequently, the attacker can analyze the cache state via a side-channel to recover the sensitive data that was transiently accessed using the incorrect forwarding.

### Scenario 2
An attacker is able to run code within a restricted sandbox. Since some processors try to predict when a store will forward data to a subsequent load, the attacker trains the CPU's memory predictor to believe that a specific load instruction depends on a previous store instruction. In the next step, the attacker executes the code in a state where the dependency does not exist, but the predictor still speculates that it does. The CPU transiently forwards stale data to the load instruction, effectively bypassing software bounds checking. The attacker uses this transient window to leak data from outside the sandbox by encoding the stale value into the cache hierarchy, which is retrieved via timing analysis.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | The hardware designer can attempt to prevent transient execution from causing observable discrepancies in specific covert channels. | Limited |
| Requirements | - | Processor designers, system software vendors, or other agents may choose to restrict the ability of unprivileged software to access to high-resolution timers that are commonly used to monitor covert channels. | Defense in Depth |
| Requirements | - | Processor designers may expose instructions or other architectural features that allow software to mitigate the effects of transient execution, but without disabling predictors. | Moderate |
| Requirements | - | Processor designers may expose registers (for example, control registers or model-specific registers) that allow privileged and/or user software to disable specific predictors or other hardware features that can cause confidential data to be exposed during transient execution. | Limited |
| Build and Compilation | - | Use software techniques (including the use of serialization instructions) that are intended to reduce the number of instructions that can be executed transiently after a processor event or misprediction. | Incidental |
| Build and Compilation | - | Isolate sandboxes or managed runtimes in separate address spaces (separate processes). | High |
| Build and Compilation | - | Include serialization instructions (for example, LFENCE) that prevent processor events or mis-predictions prior to the serialization instruction from causing transient execution after the serialization instruction. | Moderate |
| Build and Compilation | - | Use software techniques that can mitigate the consequences of transient execution. For example, address masking can be used in some circumstances to prevent out-of-bounds transient reads. | Limited |
| Build and Compilation | - | If the weakness is exposed by a single instruction (or a small set of instructions), then the compiler (or JIT, etc.) can be configured to prevent the affected instruction(s) from being generated. | Limited |
| Documentation | - | If a hardware feature can allow incorrect or stale data to be forwarded to transient operations, the hardware designer may opt to disclose this behavior in architecture documentation. | High |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design store-to-load forwarding logic to enforce that forwarded data respects architectural permission checks, preventing stale or incorrect data from being used by transient loads | Critical |
| Architecture and Design | Implement store buffer partitioning between security domains to prevent cross-domain data forwarding during speculative execution | Critical |
| Build and Compilation | Insert serialization barriers (LFENCE or equivalent) between store and load operations in security-critical code paths where incorrect forwarding could leak sensitive data | High |
| Build and Compilation | Isolate untrusted sandboxes and managed runtimes into separate address spaces to limit the impact of speculative data forwarding attacks | High |
| Testing | Develop and execute test cases that specifically target store-to-load forwarding behavior under fault conditions, verifying that stale data cannot be observed through cache timing side channels | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
