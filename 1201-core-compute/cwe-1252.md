# CWE-1252 - CPU Hardware Not Configured to Support Exclusivity of Write and Execute Operations
## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can execute unauthorized code and commands |
| **Integrity** | The attacker can execute unauthorized code and commands |
| **Availability** | Low impact (An availability loss may be a result of integrity loss, but not a direct consequence of the CWE) |

## Attack Scenario

An attacker is able to corrupt a low trust application via a low trust interface and load a malicious program into memory. In a normal case, and more generally in the majority of recent hardware, systems use a special bit to distinguish between memory types (instructions, which can be executed, versus data, which should not). However, in this scenario, the CPU lacks an MMU/MPU (Memory Management Unit / Memory Protection Unit). This lack of hardware prevents the CPU from distinguishing if the memory content is an instruction or data, as it lacks the hardware gate to refuse execution. Consequently, the attacker can trigger the execution of the injected data, leading to the execution of malicious code and allowing the attacker to gain full control over the system.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Implement a dedicated bit that can be leveraged by the Operating System to mark data areas as non-executable. If such a bit is not available in the CPU, implement MMU/MPU (memory management unit / memory protection unit). | - |
| Integration | - | If MMU/MPU are not available, then the firewalls need to be implemented in the SoC interconnect to mimic the write-exclusivity operation. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement hardware-enforced W^X (Write XOR Execute) policy via an NX/XN bit in the MMU page table entries, ensuring no memory region is simultaneously writable and executable | Critical |
| Architecture and Design | Include an MPU with at least 8 configurable regions supporting independent read, write, and execute permissions for both privileged and unprivileged modes | Critical |
| Integration | Configure SoC interconnect firewalls to enforce memory region permissions when MPU/MMU is absent, blocking execution from SRAM/DRAM regions designated for data storage | High |
| Testing | Validate W^X enforcement through penetration testing that attempts code injection and execution from data regions across all privilege levels | High |
| Patching and Maintenance | Ensure firmware update mechanisms cannot be abused to load executable content into data-only regions by verifying code signatures before mapping as executable | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
