# CWE-1423 - Exposure of Sensitive Information caused by Shared Microarchitectural Predictor State that Influences Transient Execution


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2017-5754](https://www.cve.org/CVERecord?id=CVE-2017-5754) | (Branch Target Injection, BTI, Spectre v2). Shared
					microarchitectural indirect branch predictor state may allow code to
					influence transient execution across a process, VM, or privilege
					boundary, potentially exposing data that is accessible beyond the
					boundary. |
| [CVE-2022-0001](https://www.cve.org/CVERecord?id=CVE-2022-0001) | (Branch History Injection, BHI, Spectre-BHB). Shared
					branch history state may allow user-mode code to influence transient
					execution in the kernel, potentially exposing kernel data over a
					covert channel. |
| [CVE-2021-33149](https://www.cve.org/CVERecord?id=CVE-2021-33149) | (RSB underflow, Retbleed). Shared return stack buffer
					state may allow code that executes before a prediction barrier to
					influence transient execution after the prediction barrier,
					potentially exposing data that is accessible beyond the barrier over a
					covert channel. |

## Attack Scenarios

### Scenario 1
An attacker is able to execute code on a hardware thread while a victim executes on another thread of the same physical core (simultaneous multithreading). In the next step, the attacker manipulates the shared microarchitectural predictor state (such as the branch prediction history) to influence the victim's execution. Via this shared state, the attacker causes the victim to perform a transient execution of a gadget, exposing sensitive data via a covert channel (such as cache timing analysis) that is accessible within the attacker's processor context.

### Scenario 2
An attacker is able to run a process in user mode and train the microarchitectural predictor with specific patterns. In the next step, the attacker triggers a transition to a higher privilege level (e.g., via a system call to the kernel) across a hardware boundary. Consequently, the privileged code uses the attacker-influenced predictor state, leading to transient execution that accesses private data, which the attacker can then infer by monitoring observable discrepancies in the shared resources.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | The hardware designer can attempt to prevent transient execution from causing observable discrepancies in specific covert channels. | - |
| Architecture and Design | - | Hardware designers may choose to use microarchitectural bits to tag predictor entries (e.g., kernel-mode bit, SMT thread ID) to enforce that predictions must have been trained in the current mode or thread. | Moderate |
| Architecture and Design | - | Hardware designers may choose to sanitize microarchitectural predictor state (for example, branch prediction history) when the processor transitions to a different context, or expose instruction(s) that allow software to sanitize predictor state. | Moderate |
| Implementation | - | System software can mitigate this weakness by invoking predictor-state-sanitizing operations (for example, the indirect branch prediction barrier on Intel x86) when switching from one context to another. | Moderate |
| Build and Compilation | - | If the weakness is exposed by a single instruction (or a small set of instructions), then the compiler (or JIT, etc.) can be configured to prevent the affected instruction(s) from being generated. One prominent example is retpoline. | Limited |
| Build and Compilation | - | Use control-flow integrity (CFI) techniques to constrain the behavior of instructions that redirect the instruction pointer, such as indirect branch instructions. | Moderate |
| Build and Compilation | - | Use software techniques (including the use of serialization instructions) that are intended to reduce the number of instructions that can be executed transiently after a processor event or misprediction. | Incidental |
| System Configuration | - | Some systems may allow the user to disable predictor sharing (e.g., via BIOS configuration or model-specific registers). | Moderate |
| Patching and Maintenance | - | The hardware vendor may provide a patch to sanitize predictor state when the processor transitions to a different context, or to prevent predictor entries from being shared across SMT threads. | Moderate |
| Documentation | - | If a hardware feature can allow microarchitectural predictor state to be shared between contexts, SMT threads, or other architecturally defined boundaries, the hardware designer may opt to disclose this behavior in architecture documentation. | High |
| Requirements | - | Processor designers, system software vendors, or other agents may choose to restrict the ability of unprivileged software to access to high-resolution timers that are commonly used to monitor covert channels. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement predictor entry tagging with context identifiers (privilege level, ASID, SMT thread ID) so that predictor entries trained in one context cannot influence predictions in another context | Critical |
| Architecture and Design | Design hardware to automatically flush or invalidate branch predictor state (BTB, PHT, RSB) on privilege transitions (syscall/sysret, interrupt, VM entry/exit) with minimal performance overhead | Critical |
| Implementation | Deploy indirect branch prediction barriers (IBPB) and single-thread indirect branch predictors (STIBP) on context switches and privilege transitions in OS and hypervisor software | High |
| Build and Compilation | Compile security-sensitive software with retpoline or equivalent indirect branch mitigation techniques, and enable CFI to limit gadget availability | High |
| System Configuration | On security-critical systems, enable predictor isolation features (IBRS, STIBP, or equivalent) and consider disabling SMT where cross-thread predictor poisoning poses unacceptable risk | Medium |

*These remediations are recommended best practices and are not sourced from MITRE.*
