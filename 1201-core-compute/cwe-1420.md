# CWE-1420 - Exposure of Sensitive Information during Transient Execution


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2017-5753](https://www.cve.org/CVERecord?id=CVE-2017-5753) | Microarchitectural conditional branch predictors may allow operations to execute transiently after a misprediction, potentially exposing data over a covert channel. |
| [CVE-2021-0089](https://www.cve.org/CVERecord?id=CVE-2021-0089) | A machine clear triggered by self-modifying code may allow incorrect operations to execute transiently, potentially exposing data over a covert channel. |
| [CVE-2022-0002](https://www.cve.org/CVERecord?id=CVE-2022-0002) | Microarchitectural indirect branch predictors may allow incorrect operations to execute transiently after a misprediction, potentially exposing data over a covert channel. |

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can extract confidential data and read memory |
| **Integrity** | Low impact |
| **Availability** | Low impact |

## Attack Scenarios

### Scenario 1
An attacker is able to corrupt a low trust application from a low trust interface. In the next step, the attacker is able to execute malicious code on the CPU. Via this code, the attacker is able to trigger transient execution and use the transient execution weakness to expose data via a covert channel (using timing analysis for e.g.) that is accessible within an attacker's processor context.

### Scenario 2
An attacker is able to identify a vulnerable code sequence in a high trust application. After this, the attacker uses the vulnerable code (gadget) while triggering the transient execution (via a Page Fault, a General Protection Fault, a manipulation of Branch Prediction system or Microcode Assists for e.g.) to perform a transient execution to expose data via a covert channel (using timing analysis for e.g.) that is accessible within the victim's processor context, via Out of Bounds (forces the CPU to check a data outside of the normal value expected by the program, to expose a close data in term of memory address) for example.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | The hardware designer can attempt to prevent transient execution from causing observable discrepancies in specific covert channels. | Limited |
| Requirements | - | Processor designers may expose instructions or other architectural features that allow software to mitigate the effects of transient execution, but without disabling predictors. These features may also help to limit opportunities for data exposure. | Moderate |
| Requirements | - | Processor designers may expose registers (for example, control registers or model-specific registers) that allow privileged and/or user software to disable specific predictors or other hardware features that can cause confidential data to be exposed during transient execution. | Limited |
| Requirements | - | Processor designers, system software vendors, or other agents may choose to restrict the ability of unprivileged software to access to high-resolution timers that are commonly used to monitor covert channels. | Defense in Depth |
| Build and Compilation | - | Isolate sandboxes or managed runtimes in separate address spaces (separate processes). | High |
| Build and Compilation | - | Include serialization instructions (for example, LFENCE) that prevent processor events or mis-predictions prior to the serialization instruction from causing transient execution after the serialization instruction. | Moderate |
| Build and Compilation | - | Use control-flow integrity (CFI) techniques to constrain the behavior of instructions that redirect the instruction pointer, such as indirect branch instructions. | Moderate |
| Build and Compilation | - | If the weakness is exposed by a single instruction (or a small set of instructions), then the compiler (or JIT, etc.) can be configured to prevent the affected instruction(s) from being generated, and instead generate an alternate sequence of instructions that is not affected by the weakness (e.g., retpoline). | Limited |
| Build and Compilation | - | Use software techniques that can mitigate the consequences of transient execution. For example, address masking can be used in some circumstances to prevent out-of-bounds transient reads. | Limited |
| Build and Compilation | - | Use software techniques (including the use of serialization instructions) that are intended to reduce the number of instructions that can be executed transiently after a processor event or misprediction. | Incidental |
| Documentation | - | If a hardware feature can allow incorrect operations (or correct operations with incorrect data) to execute transiently, the hardware designer may opt to disclose this behavior in architecture documentation. | High |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design speculation barriers into the processor pipeline that prevent transient instructions from accessing data outside the current privilege domain, ensuring microarchitectural state changes from mispredicted paths are not observable | Critical |
| Architecture and Design | Implement hardware partitioning of branch prediction structures (BTB, PHT, RSB) across security domains to prevent cross-domain training attacks | Critical |
| Build and Compilation | Deploy serialization barriers (LFENCE or equivalent) after conditional branches that guard security-sensitive memory accesses, and use retpoline or equivalent for indirect branches | High |
| Requirements | Expose MSRs or control registers that allow the OS/hypervisor to selectively enable/disable speculative features per security domain, balancing performance and security | High |
| Testing | Validate transient execution mitigations using proof-of-concept exploits for known Spectre variants across all supported privilege boundaries (user/kernel, guest/host, enclave/non-enclave) | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
