# CWE-1274 - Improper Access Control for Volatile Memory Containing Boot Code

## CWE Explanation

This weakness occurs during the secure boot process when the system copies authenticated bootloader code from slow non-volatile storage (like SPI flash) into faster volatile memory (like SRAM or DRAM) for execution, but fails to properly restrict write access to that volatile memory region.

Secure boot relies on a chain of trust where each stage verifies the next. However, simply verifying the digital signature of the code as it sits in flash is insufficient if the copy in RAM can be modified. If the memory region holding the active bootloader is not locked down (e.g., marked read-only or protected by a firewall/MPU) immediately after the code is copied and verified, an attacker can modify the instructions in memory. This effectively allows the execution of malicious code that appears to have been authenticated, breaking the chain of trust.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Low impact (*Primary risk is integrity/execution, though secrets could be exfiltrated later*) |
| **Integrity** | The attacker can overwrite the authenticated bootloader in memory |
| **Availability** | The attacker can render the device unbootable or unstable |

## Attack Scenario

An attacker has the ability to write to system memory, perhaps through a DMA-capable peripheral or a compromised secondary processor core. In the next step, the target system performs its secure boot sequence. The Read-Only Memory (ROM) code verifies the signature of the next-stage bootloader stored in external flash and copies it into internal SRAM for faster execution. However, the system fails to immediately configure the memory controller or firewall to mark this SRAM region as "read-only" or "privileged-access only." Capitalizing on this missing protection, the attacker overwrites the validated bootloader instructions in SRAM with malicious shellcode just before the ROM passes execution control to it. This results in the processor executing the attacker's code with the privileges of the bootloader, completely bypassing the signature check that just occurred.

## MITRE Reference Data

### Description

The product conducts a secure-boot process that transfers bootloader code from Non-Volatile Memory (NVM) into Volatile Memory (VM), but it does not have sufficient access control or other protections for the Volatile Memory.

### Extended Description

Adversaries could bypass the secure-boot process and execute their own untrusted, malicious boot code.

As a part of a secure-boot process, the read-only-memory (ROM) code for a System-on-Chip (SoC) or other system fetches bootloader code from Non-Volatile Memory (NVM) and stores the code in Volatile Memory (VM), such as dynamic, random-access memory (DRAM) or static, random-access memory (SRAM). The NVM is usually external to the SoC, while the VM is internal to the SoC. As the code is transferred from NVM to VM, it is authenticated by the SoC's ROM code.

A typical SoC secure boot's flow includes fetching the next piece of code (i.e., the boot loader) from NVM (e.g., serial, peripheral interface (SPI) flash), and transferring it to DRAM/SRAM volatile, internal memory, which is more efficient.
