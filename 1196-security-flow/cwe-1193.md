## CWE Explanation

This weakness occurs when a System-on-Chip (SoC) powers on and enables secondary execution cores or peripheral microcontrollers running untrusted firmware before the system's global security controls are active. Modern SoCs often contain multiple processing units (e.g., for graphics, audio, or connectivity) that share the system bus and memory.

If these secondary units are allowed to execute code and issue bus transactions before the primary trusted firmware (Root of Trust) has configured the fabric access controls—such as firewalls, memory protection units, or IOMMUs—they operate in a promiscuous environment. An attacker who has compromised the firmware of such a secondary core can exploit this window to access restricted memory regions, modify the boot process, or exfiltrate secrets that should be protected by the yet-to-be-configured access controls.

The fundamental issue is an incorrect dependency in the boot sequence: the "trust boundaries" are established after the potential "threat actors" (untrusted cores) are already active.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read sensitive boot secrets or keys from shared memory |
| **Integrity** | The attacker can overwrite the main system bootloader or trusted firmware |
| **Availability** | Low impact (*System instability possible if shared resources are corrupted*) |

## Attack Scenario

An attacker gains control over the firmware of a secondary, untrusted core within the SoC (e.g., a sensor hub or radio controller). In the next step, the system resets and begins its boot sequence. The hardware logic powers on this secondary core early in the process, allowing it to begin executing instructions and issuing bus transactions immediately. Crucially, this happens before the primary trusted core has finished running its initialization code to configure the system fabric's access control lists and memory firewalls. Due to this race condition in the boot flow, the attacker-controlled core exploits the window of unprotected access to read from or write to reserved memory regions used by the primary trusted core. This results in a full compromise of the system's chain of trust before the security mechanisms are even instantiated.

## MITRE Reference Data

### Description

The product enables components that contain untrusted firmware before memory and fabric access controls have been enabled.

### Extended Description

After initial reset, System-on-Chip (SoC) fabric access controls and other security features need to be programmed by trusted firmware as part of the boot sequence. If untrusted IPs or peripheral microcontrollers are enabled first, then the untrusted component can master transactions on the hardware bus and target memory or other assets to compromise the SoC boot firmware.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design the SoC power-on reset sequence so that fabric firewalls and memory protection units are fully programmed by the Root of Trust before any untrusted execution core is released from reset. | Critical |
| Architecture and Design | Implement hardware reset-hold signals for secondary cores that can only be deasserted by authenticated trusted firmware after access controls are configured. | Critical |
| Implementation | Ensure that all bus master IDs for untrusted cores are mapped to a default-deny policy in the fabric firewall until explicitly granted access by the trusted boot firmware. | High |
| Testing | Validate through fault injection and boot-sequence testing that no untrusted core can issue bus transactions before fabric access controls are active, including under reset glitch scenarios. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
