## CWE Explanation

This weakness occurs when a system enables a Direct Memory Access (DMA) capable device during the boot process before the necessary security controls, such as an Input-Output Memory Management Unit (IOMMU), are fully configured and active. DMA allows hardware peripherals to read from and write directly to main memory, bypassing the central processor's supervision to improve performance.

When these devices are powered on and enabled early in the boot sequence—prior to the operating system or firmware establishing memory protection ranges or virtualization barriers—they operate with unrestricted access to physical memory. This creates a window of vulnerability where a malicious or compromised peripheral can access sensitive data or modify critical system code that is being loaded, effectively bypassing the intended security architecture.

The root cause is a race condition between the enablement of the peripheral's DMA capabilities and the initialization of the security mechanisms designed to constrain those capabilities.

| **Availability** | Low impact (*System instability is possible due to corrupted memory*) |

## Attack Scenario

An attacker is able to interact with a DMA-capable peripheral (such as a PCIe device) connected to the target system. In the next step, the attacker triggers a DMA transaction during the early boot phase, specifically after the device is powered on but before the IOMMU or other fabric access controls are fully initialized by the boot firmware. Due to the premature enablement of the DMA device without active protections, the transaction is allowed to proceed unchecked. This results in the attacker successfully reading or writing to arbitrary physical memory locations, potentially injecting malicious code into the booting kernel or extracting cryptographic keys.

## MITRE Reference Data

### Description

The product enables a Direct Memory Access (DMA) capable device before the security configuration settings are established, which allows an attacker to extract data from or gain privileges on the product.

### Extended Description

DMA is included in a number of devices because it allows data transfer between the computer and the connected device, using direct hardware access to read or write directly to main memory without any OS interaction. An attacker could exploit this to access secrets. Several virtualization-based mitigations have been introduced to thwart DMA attacks. These are usually configured/setup during boot time. However, certain IPs that are powered up before boot is complete (known as early boot IPs) may be DMA capable. Such IPs, if not trusted, could launch DMA attacks and gain access to assets that should otherwise be protected.

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Configure and enable the IOMMU or equivalent DMA protection (e.g., Intel VT-d, ARM SMMU) before powering on any DMA-capable peripheral during the boot sequence. | Critical |
| Architecture and Design | Define a strict boot ordering policy in the SoC power-on sequencer that gates DMA device enablement on the completion of fabric access control initialization. | Critical |
| Implementation | Implement hardware interlocks that prevent DMA-capable devices from issuing bus transactions until a trusted firmware handshake signals that memory protections are active. | High |
| Testing | Perform pre-silicon and post-silicon validation of the boot sequence to verify that no DMA transactions can occur before IOMMU/firewall configuration is complete. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
