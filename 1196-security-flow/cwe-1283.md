## CWE Explanation

This weakness involves the improper protection of hardware registers used to store security measurements, such as the hashes of bootloader code, firmware, or operating system components. In a measured boot process, the system calculates a cryptographic hash of each component before executing it and "extends" this value into a special register (like a Platform Configuration Register or PCR). This aggregate measurement serves as the root of trust for remote attestation, proving to an external party that the system is in a known, trusted state.

If these reporting registers are mutable—meaning they can be overwritten or reset by software after the boot process has completed—the integrity of the entire attestation mechanism is compromised. An attacker who has compromised the system can simply overwrite the registers with "golden" values corresponding to a healthy system, effectively hiding their presence and allowing the compromised device to authenticate as trusted.

## Attack Scenario

An attacker has successfully infected the operating system or hypervisor with malware. In the next step, the attacker attempts to bypass the system's remote attestation check, which is designed to detect such compromises by verifying the boot chain measurements stored in hardware registers. The attacker reads the expected "good" hash values from a known clean system. Then, targeting the measurement registers which lack sufficient write-protection (e.g., they are not "write-once" or "read-only" after boot), the attacker overwrites the current (compromised) measurement values with the expected "good" values. This results in the hardware generating a valid digital signature over the falsified measurements, fooling the remote verifier into accepting the compromised system as healthy and trusted.

## MITRE Reference Data

### Description

The register contents used for attestation or measurement reporting data to verify boot flow are modifiable by an adversary.

### Extended Description

A System-on-Chip (SoC) implements secure boot or verified boot. During this boot flow, the SoC often measures the code that it authenticates. The measurement is usually done by calculating the one-way hash of the code binary and extending it to the previous hash. The hashing algorithm should be a Secure One-Way hash function. The final hash, i.e., the value obtained after the completion of the boot flow, serves as the measurement data used in reporting or in attestation. The calculated hash is often stored in registers that can later be read by the party of interest to determine tampering of the boot flow. A common weakness is that the contents in these registers are modifiable by an adversary, thus spoofing the measurement.

### Demonstrative Examples

#### Example 1

The SoC extends the hash and stores the results in registers. Without protection, an adversary can write their chosen hash values to these registers. Thus, the attacker controls the reported results.

To prevent the above scenario, the registers should have one or more of the following properties:

*   Should be Read-Only with respect to an adversary
*   Cannot be extended or modifiable either directly or indirectly (using a trusted agent as proxy) by an adversary
*   Should have appropriate access controls or protections

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | — | Measurement data should be stored in registers that are read-only or otherwise have access controls that prevent modification by an untrusted agent. | — |

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Store all attestation measurement values in hardware registers that support only the "extend" operation (hash-extend) and cannot be directly written or reset by software after the boot process initializes them. | Critical |
| Architecture and Design | Implement hardware-enforced write-once or append-only semantics for PCR-equivalent registers, ensuring that no software privilege level (including root/kernel) can clear or overwrite measurement values until a full platform reset. | Critical |
| Implementation | Protect measurement register access with hardware access control policies tied to the boot phase, so that only the trusted ROM or first-stage bootloader can initialize and extend these registers. | High |
| Testing | Validate through pre-silicon formal verification and post-silicon penetration testing that measurement registers cannot be modified, reset, or bypassed through any software or debug interface after boot. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
