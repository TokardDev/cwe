# CWE-1328 - Security Version Number Mutable to Older Versions
## CWE Explanation

This weakness involves the improper implementation of anti-rollback mechanisms in hardware, specifically the storage of the Security Version Number (SVN). To prevent attackers from downgrading a system to an older, vulnerable firmware version, devices typically store a monotonic counter or SVN in persistent storage. During a firmware update, the system checks that the new firmware's version is greater than or equal to the stored SVN.

If this stored SVN is mutable—meaning it can be decremented, reset, or overwritten with a lower value by an unauthorized entity—the anti-rollback protection fails. An attacker can forcibly lower the stored version number and then flash an outdated, vulnerable firmware image. Once the vulnerable firmware is running, the attacker can exploit known bugs that had been patched in newer versions. Conversely, if an attacker can arbitrarily *increment* the SVN to a maximum value, they can permanently prevent the device from accepting legitimate future updates (Denial of Service).

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Low impact (*Indirectly compromised if rolled-back firmware has leaks*) |
| **Integrity** | The attacker can install older, vulnerable firmware versions |
| **Availability** | The attacker can prevent future updates by maximizing the counter (DoS) |

## Attack Scenario

An attacker identifies a critical buffer overflow vulnerability in version 1.0 of a device's firmware, which the vendor has patched in version 2.0. The device is currently running version 2.0. The attacker gains local access and targets the hardware fuses or battery-backed registers where the Anti-Rollback Security Version Number (SVN) is stored. Due to a flaw in the hardware access control logic, the attacker is able to execute a command that resets or decrements the stored SVN from '2' back to '1'. With the hardware constraint removed, the attacker flashes the vulnerable version 1.0 firmware onto the device. The secure boot process validates the signature (which is technically valid) and accepts the version number (which now matches the lowered hardware SVN), allowing the attacker to regain a foothold via the old vulnerability.

## MITRE Reference Data

### Description

Security-version number in hardware is mutable, resulting in the ability to downgrade (roll-back) the boot firmware to vulnerable code versions.

### Extended Description

A System-on-Chip (SoC) implements secure boot or verified boot. It might support a security version number, which prevents downgrading the current firmware to a vulnerable version. Once downgraded to a previous version, an adversary can launch exploits on the SoC and thus compromise the security of the SoC. These downgrade attacks are also referred to as roll-back attacks.

The security version number must be stored securely and persistently across power-on resets. A common weakness is that the security version number is modifiable by an adversary, allowing roll-back or downgrade attacks or, under certain circumstances, preventing upgrades (i.e. Denial-of-Service on upgrades). In both cases, the SoC is in a vulnerable state.

### Demonstrative Examples

#### Example 1

A new version of firmware is signed with a security version number higher than the previous version. During the firmware update process the SoC checks for the security version number and upgrades the SoC firmware with the latest version. This security version number is stored in persistent memory upon successful upgrade for use across power-on resets.

In general, if the security version number is mutable, the implementation is vulnerable. A mutable security version number allows an adversary to change the security version to a lower value to allow roll-back or to a higher value to prevent future upgrades.

The security version number should be stored in immutable hardware such as fuses, and the writes to these fuses should be highly access-controlled with appropriate authentication and authorization protections.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | — | When architecting the system, security version data should be designated for storage in registers that are either read-only or have access controls that prevent modification by an untrusted agent. | — |
| Implementation | — | During implementation and test, security version data should be demonstrated to be read-only and access controls should be validated. | — |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Store the Security Version Number in OTP fuses with a monotonic counter design (e.g., thermometer-coded fuse bank) that physically prevents decrementing the value. | Critical |
| Architecture and Design | Restrict the fuse programming interface for the SVN to only the authenticated secure boot firmware, ensuring no software or debug interface can modify the counter without cryptographic authorization. | Critical |
| Implementation | Implement an upper-bound check on SVN increments to prevent an attacker from maximizing the counter value and causing a denial-of-service on future firmware updates. | High |
| Testing | Validate anti-rollback protection in post-silicon testing by attempting to flash older firmware versions and confirming the boot process rejects them based on the stored SVN. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
