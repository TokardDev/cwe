# CWE-1280 - Access Control Check Implemented After Asset is Accessed

## CWE Explanation

This weakness occurs in hardware designs when the logic responsible for granting access to a protected asset operates in parallel with or subsequent to the logic that performs the access itself, rather than preceding it.

In synchronous hardware design, operations must be strictly ordered to ensure security. If an "Access Enable" signal is generated in the same clock cycle as the "Data Write" operation but, due to propagation delays or incorrect procedural ordering in HDL, the write occurs before the enable signal is stable or evaluated, the protection is bypassed. This often happens when developers use blocking assignments improperly in Verilog or fail to account for the latency of the authorization logic.

The result is a "glitch" or a transient window where unauthorized data is successfully written to a register or read from memory before the security mechanism has a chance to block it. Even if the access is blocked in the *next* cycle, the damage (data corruption or leakage) is already done.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An attacker can read sensitive data during the transient window before the read access is officially denied. |
| **Integrity** | An attacker can successfully write to a protected register because the write operation completes before the access check logic inhibits it. |
| **Availability** | Low impact, though incorrect state transitions could potentially lead to instability. |

## Attack Scenario

A hardware register storing a root cryptographic key is protected by an access control module. The design intends that only the Security Coprocessor (ID `0x4`) can write to this register. However, the Verilog code implementing the write logic is flawed. Inside the clocked process, the data update logic (`key_reg = wdata`) is written before the permission check logic (`access_granted = (id == 0x4)`). Due to the semantics of the hardware description language (or signal race conditions in the synthesized logic), the register accepts the new value from the data bus *before* the `access_granted` signal effectively blocks the write. An attacker running on the main CPU (ID `0x1`) initiates a write to the key register. Although the system eventually flags a "Permission Denied" error, the `key_reg` has already been overwritten with the attacker's known value, compromising the entire cryptographic chain.

## MITRE Reference Data

### Description

A product's hardware-based access control check occurs after the asset has been accessed.

### Extended Description

The product implements a hardware-based access control check. The asset should be accessible only after the check is successful. If, however, this operation is not atomic and the asset is accessed before the check is complete, the security of the system may be compromised.

### Demonstrative Examples

#### Example 1

Assume that the module foo_bar implements a protected register. The register content is the asset. Only transactions made by user id (indicated by signal usr_id) 0x4 are allowed to modify the register contents. The signal grant_access is used to provide access.

```verilog
(bad code)
module foo_bar(data_out, usr_id, data_in, clk, rst_n);
    output reg [7:0] data_out;
    input wire [2:0] usr_id;
    input wire [7:0] data_in;
    input wire clk, rst_n;
    wire grant_access;
    always @ (posedge clk or negedge rst_n)
    begin
        if (!rst_n)
            data_out = 0;
        else
            data_out = (grant_access) ? data_in : data_out;
            assign grant_access = (usr_id == 3'h4) ? 1'b1 : 1'b0;
    end
endmodule
```

This code uses Verilog blocking assignments for data_out and grant_access. Therefore, these assignments happen sequentially (i.e., data_out is updated to new value first, and grant_access is updated the next cycle) and not in parallel. Therefore, the asset data_out is allowed to be modified even before the access control check is complete and grant_access signal is set. Since grant_access does not have a reset value, it will be meta-stable and will randomly go to either 0 or 1.

Flipping the order of the assignment of data_out and grant_access should solve the problem. The correct snippet of code is shown below.

```verilog
(good code)
always @ (posedge clk or negedge rst_n)
begin
    if (!rst_n)
        data_out = 0;
    else
        assign grant_access = (usr_id == 3'h4) ? 1'b1 : 1'b0;
        data_out = (grant_access) ? data_in : data_out;
end
endmodule
```
