## CWE Explanation

This weakness occurs when a system, component, or device is initialized with default permissions that are overly permissive, granting unauthorized actors access to sensitive resources. In a hardware context, this often manifests as registers, memory regions, or debug interfaces being readable or writable by lower-privileged software or external interfaces by default upon reset or power-on.

When default permissions are too loose, an attacker can access sensitive data or modify system configuration without needing to bypass authentication mechanisms or exploit complex vulnerabilities. They simply utilize the access rights that were implicitly granted by the insecure default configuration.

The root cause is typically an oversight in the design or implementation phase where the principle of least privilege is not applied to the initial state of the system. Developers may choose permissive defaults to ensure ease of use or interoperability during initial bring-up, failing to lock them down for the production release.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read sensitive configuration data or secrets stored in registers or memory. |
| **Integrity** | The attacker can modify critical system settings, potentially disabling security features or altering device behavior. |
| **Availability** | The attacker can modify configuration registers to crash the system or render it unusable. |

## Attack Scenario

An attacker is able to execute code on the device with low privileges or access a physical interface. Upon system reset, a critical hardware control register responsible for defining memory protection regions defaults to being writable by any bus master. In the next step, the attacker writes to this register to disable memory protection for a secure memory region. Since the default permission allowed this write operation, the attacker successfully disables the protection and can now read or modify the secure boot code or cryptographic keys stored in that region.

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design; Operation | - | The architecture needs to access and modification attributes for files to only those users who actually require those actions. | - |
| Architecture and Design | Separation of Privilege | Compartmentalize the system to have "safe" areas where trust boundaries can be unambiguously drawn. Do not allow sensitive data to go outside of the trust boundary and always be careful when interfacing with a compartment outside of the safe area. Ensure that appropriate compartmentalization is built into the system design, and the compartmentalization allows for and reinforces privilege separation functionality. Architects and designers should rely on the principle of least privilege to decide the appropriate time to use privileges and the time to drop privileges. | - |

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Apply the principle of least privilege to all hardware register and memory region defaults: all resources should be inaccessible (read-protected and write-protected) at power-on reset unless explicitly opened by trusted firmware. | Critical |
| Implementation | Ensure that all security-sensitive registers (key storage, protection configuration, debug interfaces) default to the most restrictive permission state upon reset, requiring explicit enablement by authenticated firmware. | Critical |
| Architecture and Design | Implement a secure boot flow that configures and locks down all access control policies before any untrusted code executes, preventing exploitation of permissive defaults. | High |
| Testing | Audit all register reset values and memory region default permissions to confirm that no sensitive resource is accessible by unprivileged agents in the default state. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
