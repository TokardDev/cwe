## CWE Explanation

This weakness, often known as the "Confused Deputy" problem, occurs when a trusted component (the deputy) accepts requests from an untrusted component and forwards them to a protected resource without preserving the identity or privileges of the original requester. In hardware designs, this frequently happens with Direct Memory Access (DMA) controllers, accelerators, or bridges that operate with high privileges.

When a lower-privileged master (like a microcontroller running untrusted code) cannot directly access a protected resource (like a secure memory region or configuration register), it may instruct a higher-privileged intermediary to perform the access on its behalf. If the intermediary does not validate that the original requester has the necessary permissions, or if it performs the access using its own high-level credentials, security controls are bypassed.

The root cause is a failure to propagate security attributes (such as the Source ID or Privilege Level) across the transaction chain, or the intermediary explicitly replacing the original security attributes with its own trusted identity.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | The attacker can read sensitive data from protected memory regions via the confused deputy. |
| **Integrity** | The attacker can modify secure configuration registers or corrupt protected memory. |
| **Availability** | The attacker can overwrite critical system structures, causing a denial of service. |

## Attack Scenario

An attacker is able to execute code with low privileges (e.g., Ring 3) on a microcontroller within a System-on-Chip (SoC). Direct access from this microcontroller to a secure IP core containing design secrets is blocked by hardware access controls. In the next step, the attacker configures a Direct Memory Access (DMA) controller, which is a trusted bus master, to transfer data from the secure IP core to a non-secure memory region readable by the attacker. The DMA controller initiates the read transaction using its own privileged identity rather than forwarding the identity of the requesting microcontroller. Since the DMA controller is trusted, the access control logic permits the read, allowing the attacker to bypass the intended restriction and extract the secrets.

## MITRE Reference Data

### Description

The product receives a request, message, or directive from an upstream component, but the product does not sufficiently preserve the original source of the request before forwarding the request to an external actor that is outside of the product's control sphere. This causes the product to appear to be the source of the request, leading it to act as a proxy or other intermediary between the upstream component and the external actor.

### Extended Description

If an attacker cannot directly contact a target, but the product has access to the target, then the attacker can send a request to the product and have it be forwarded to the target. The request would appear to be coming from the product's system, not the attacker's system. As a result, the attacker can bypass access controls (such as firewalls) or hide the source of malicious requests, since the requests would not be coming directly from the attacker.

Since proxy functionality and message-forwarding often serve a legitimate purpose, this issue only becomes a vulnerability when:

*   The product runs with different privileges or on a different system, or otherwise has different levels of access than the upstream component;
*   The attacker is prevented from making the request directly to the target; and
*   The attacker can create a request that the proxy does not explicitly intend to be forwarded on the behalf of the requester. Such a request might point to an unexpected hostname, port number, hardware IP, or service. Or, the request might be sent to an allowed service, but the request could contain disallowed directives, commands, or resources.

### Demonstrative Examples

#### Example 1

A SoC contains a microcontroller (running ring-3 (least trusted ring) code), a Memory Mapped Input Output (MMIO) mapped IP core (containing design-house secrets), and a Direct Memory Access (DMA) controller, among several other compute elements and peripherals. The SoC implements access control to protect the registers in the IP core (which registers store the design-house secrets) from malicious, ring-3 (least trusted ring) code executing on the microcontroller. The DMA controller, however, is not blocked off from accessing the IP core for functional reasons.

```other
(bad code)
The code in ring-3 (least trusted ring) of the
microcontroller attempts to directly read the protected
registers in IP core through MMIO transactions. However,
this attempt is blocked due to the implemented access
control. Now, the microcontroller configures the DMA core
to transfer data from the protected registers to a memory
region that it has access to. The DMA core, which is
acting as an intermediary in this transaction, does not
preserve the identity of the microcontroller and, instead,
initiates a new transaction with its own identity. Since
the DMA core has access, the transaction (and hence, the
attack) is successful.
```

The weakness here is that the intermediary or the proxy agent did not ensure the immutability of the identity of the microcontroller initiating the transaction.

```other
(good code)
The DMA
core forwards this transaction with the identity of the
code executing on the microcontroller, which is the
original initiator of the end-to-end transaction. Now the
transaction is blocked, as a result of forwarding the
identity of the true initiator which lacks the permission
to access the confidential MMIO mapped IP core.
```

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Enforce the use of strong mutual authentication mechanism between the two parties. | - |
| Architecture and Design | - | Whenever a product is an intermediary or proxy for transactions between two other components, the proxy core should not drop the identity of the initiator of the transaction. The immutability of the identity of the initiator must be maintained and should be forwarded all the way to the target. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Design all intermediary components (DMA controllers, bridges, accelerators) to propagate the original requester's Security Token and privilege level end-to-end, never substituting their own identity. | Critical |
| Architecture and Design | Implement IOMMU or SMMU in front of all DMA-capable masters to enforce access control based on the software context that programmed the DMA, not the DMA engine's own identity. | Critical |
| Implementation | Configure all bus bridges and proxy agents to forward the source master ID as a sideband signal; reject or flag transactions where the originator identity has been dropped or replaced. | High |
| Testing | Verify that an untrusted agent cannot use any intermediary (DMA, bridge, accelerator) to access resources that the untrusted agent is directly blocked from accessing. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
