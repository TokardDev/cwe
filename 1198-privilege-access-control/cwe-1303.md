## CWE Explanation

This weakness arises when microarchitectural resources (such as caches, branch predictors, store buffers, or execution ports) are shared between logically isolated execution contexts (e.g., processes, virtual machines, or hardware threads) without adequate partitioning or flushing mechanisms.

Modern processors optimize performance by sharing these resources to maximize utilization. However, this sharing often creates "side channels"—unintended communication paths that allow one context to infer the state or activity of another. For example, if two threads share a data cache, one thread can prime the cache and measure the access time to determine which memory lines the other thread has accessed.

This violation of architectural isolation is "non-transparent" because the shared state is not visible in the architectural state (registers, memory) but affects the timing or behavior of execution in a way that can be measured.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An attacker can extract cryptographic keys, passwords, or memory contents across security boundaries via side-channel analysis. |
| **Integrity** | In some speculative execution attacks, an attacker might influence the victim's control flow to execute gadgets that modify state (though this is less common than leakage). |
| **Availability** | An attacker can intentionally create contention for shared resources (e.g., "cache thrashing") to degrade the performance of a victim process. |

## Attack Scenario

Two processes, a Victim (handling cryptographic keys) and an Attacker, run on the same physical processor core using Simultaneous Multithreading (SMT/Hyper-threading). They share the Branch Target Buffer (BTB), a hardware structure used to predict the destination of indirect jumps. Ideally, the BTB entries should be tagged with the process ID, but the hardware implementation ignores the ID to save space. In the next step, the Attacker executes a loop of indirect jumps to train the shared BTB to predict a specific target address. When the Victim process subsequently executes an indirect jump, the hardware uses the Attacker's trained prediction to speculatively execute instructions at a gadget address chosen by the Attacker. Although the CPU eventually detects the misprediction and rolls back, the speculative execution leaves traces in the shared data cache (e.g., loading a value dependent on a secret key bit). The Attacker then probes the cache to deduce the secret key bit.

## MITRE Reference Data

### Description

Hardware structures shared across execution contexts (e.g., caches and branch predictors) can violate the expected architecture isolation between contexts.

### Extended Description

Modern processors use techniques such as out-of-order execution, speculation, prefetching, data forwarding, and caching to increase performance. Details about the implementation of these techniques are hidden from the programmer's view. This is problematic when the hardware implementation of these techniques results in resources being shared across supposedly isolated contexts. Contention for shared resources between different contexts opens covert channels that allow malicious programs executing in one context to recover information from another context.

Some examples of shared micro-architectural resources that have been used to leak information between contexts are caches, branch prediction logic, and load or store buffers. Speculative and out-of-order execution provides an attacker with increased control over which data is leaked through the covert channel.

If the extent of resource sharing between contexts in the design microarchitecture is undocumented, it is extremely difficult to ensure system assets are protected against disclosure.

### Demonstrative Examples

#### Example 1

On some processors the hardware indirect branch predictor is shared between execution contexts, for example, between sibling SMT threads. When SMT thread A executes an indirect branch to a target address X, this target may be temporarily stored by the indirect branch predictor. A subsequent indirect branch mis-prediction for SMT thread B could speculatively execute instructions at X (or at a location in B's address space that partially aliases X). Even though the processor rolls back the architectural effects of the mis-predicted indirect branch, the memory accesses alter data cache state, which is not rolled back after the indirect branch is resolved.
