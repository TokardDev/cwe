# CWE-1222 - Insufficient Granularity of Address Regions Protected by Register Locks

## CWE Explanation

This weakness occurs when a hardware protection mechanism (specifically register locks) is applied to address regions that are too large or coarsely defined, grouping together resources with conflicting access requirements.

In secure hardware designs, it is common to use "lock bits" to make critical configuration registers read-only after the boot process is complete. When a lock bit is set, any write attempts to the protected address range are blocked until the next system reset. However, if the hardware protects a large block of memory (e.g., a 1KB page) with a single lock bit, it treats all registers within that block as a single atomic unit for protection purposes.

If a designer places a register that requires runtime modification (e.g., a status flag or a dynamic control bit) within the same locking granularity region as a static security configuration register, a conflict arises. To secure the system, the lock bit must be set, but doing so unintentionally freezes the dynamic register, breaking system functionality. Conversely, leaving the lock bit cleared to allow runtime updates leaves the security configuration exposed to tampering.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | Low impact (This weakness primarily affects integrity and availability). |
| **Integrity** | An attacker can modify static security settings if the lock must be left open to support dynamic functionality. |
| **Availability** | The system may crash or fail to function correctly if a lock is set that prevents necessary runtime updates to a control register. |

## Attack Scenario

An attacker has gained code execution privileges on a running system. The target device has a security controller with a configuration space where the first 1KB is protected by a single "Lock" bit. Within this 1KB region, there is a critical `SECURE_BOOT_ENABLE` register (offset `0x0004`) that should be locked after boot, and a `SOFTWARE_MODE` register (offset `0x0310`) that legitimate software needs to write to during normal operation to switch modes. Because the hardware design groups these two registers under the same lock, the bootloader was forced to leave the "Lock" bit unset to ensure the `SOFTWARE_MODE` register remained writable. In the next step, the attacker takes advantage of this open lock to overwrite the `SECURE_BOOT_ENABLE` register, effectively disabling the device's root of trust for the next reboot sequence.

## MITRE Reference Data

### Description

The product defines a large address region protected from modification by the same register lock control bit. This results in a conflict between the functional requirement that some addresses need to be writable by software during operation and the security requirement that the system configuration lock bit must be set during the boot process.

### Extended Description

Integrated circuits and hardware IPs can expose the device configuration controls that need to be programmed after device power reset by a trusted firmware or software module (commonly set by BIOS/bootloader) and then locked from any further modification. In hardware design, this is commonly implemented using a programmable lock bit which enables/disables writing to a protected set of registers or address regions. When the programmable lock bit is set, the relevant address region can be implemented as a hardcoded value in hardware logic that cannot be changed later.

A problem can arise wherein the protected region definition is not granular enough. After the programmable lock bit has been set, then this new functionality cannot be implemented without change to the hardware design.

If a register exists within the first kilobyte address range (e.g. SW_MODE, address 0x310) and needs to be software writable at runtime, then this register cannot be written in a securely configured system since SYS_LOCK register lock bit 0 must be set to protect other security settings (e.g. SECURITY_FEATURE_ENABLE, address 0x0004). The only fix would be to change the hardware logic or not set the security lock bit.
