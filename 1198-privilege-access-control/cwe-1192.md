# CWE-1192 - Improper Identifier for IP Block used in System-On-Chip (SOC)

## CWE Explanation

This weakness occurs when Intellectual Property (IP) blocks within a System-on-Chip (SoC) are not assigned unique, immutable, and secure identifiers. In a complex SoC, multiple initiators (CPUs, DMA controllers, accelerators) interact with various targets (memories, peripherals). To enforce security policies, the system must be able to unambiguously identify the source of each transaction.

If IP blocks lack a unique identifier, or if the identifier can be spoofed or modified (e.g., programmed by software), the hardware access control logic (such as a firewall or system memory management unit) cannot correctly distinguish between trusted and untrusted agents. This failure allows malicious or malfunctioning components to impersonate trusted entities and bypass security boundaries.

The root cause is a design oversight where the identity mechanism is either missing, shared among multiple blocks, or implemented as a mutable register rather than a hardwired value tied to the hardware instantiation.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An attacker can impersonate a trusted agent to read sensitive data from protected memory. |
| **Integrity** | An attacker can bypass write protections by masquerading as a privileged component. |
| **Availability** | An attacker can disrupt system operations by initiating transactions that are falsely attributed to a critical system component. |

## Attack Scenario

An attacker gains control over a non-secure peripheral IP block (e.g., a generic DMA controller) on the SoC. The system design requires that only the Secure Boot ROM controller can write to the key storage memory. However, the SoC design does not implement hardwired, immutable Source IDs for its masters. Instead, the Source ID is configured via a software-writable register within each IP block. In the next step, the attacker reprograms the Source ID register of the compromised DMA controller to match the ID of the Secure Boot ROM controller. The attacker then initiates a write transaction to the key storage memory. The access control logic, seeing the trusted Source ID, permits the transaction, allowing the attacker to overwrite the root trust keys.

## MITRE Reference Data

### Description

The System-on-Chip (SoC) does not have unique, immutable identifiers for each of its components.

### Extended Description

A System-on-Chip (SoC) comprises several components (IP) with varied trust requirements. It is required that each IP is identified uniquely and should distinguish itself from other entities in the SoC without any ambiguity. The unique secured identity is required for various purposes. Most of the time the identity is used to route a transaction or perform certain actions, including resetting, retrieving a sensitive information, and acting upon or on behalf of something else.

There are several variants of this weakness:

*   A "missing" identifier is when the SoC does not define any mechanism to uniquely identify the IP.
*   An "insufficient" identifier might provide some defenses - for example, against the most common attacks - but it does not protect against everything that is intended.
*   A "misconfigured" mechanism occurs when a mechanism is available but not implemented correctly.
*   An "ignored" identifier occurs when the SoC/IP has not applied any policies or does not act upon the identifier securely.
