# CWE-1189 - Improper Isolation of Shared Resources on System-on-a-Chip (SoC)


## Related CVEs

| CVE | Description |
|-----|-------------|
| [CVE-2020-8698](https://www.cve.org/CVERecord?id=CVE-2020-8698) | Processor has improper isolation of shared resources allowing for information disclosure. |
| [CVE-2019-6260](https://www.cve.org/CVERecord?id=CVE-2019-6260) | Baseboard Management Controller (BMC) device implements Advanced High-performance Bus (AHB) bridges that do not require authentication for arbitrary read and write access to the BMC&#39;s physical address space from the host, and possibly the network [REF-1138]. |

## CWE Explanation

This weakness occurs in System-on-Chip (SoC) designs where resources such as memory, peripherals, or bus interconnects are shared between agents with different trust levels (e.g., a trusted security enclave and an untrusted application processor) without adequate isolation mechanisms.

In complex SoCs, sharing resources is common to optimize area and cost. For example, a single SRAM block might serve as a mailbox for communication between a secure and non-secure core, or a pin might be multiplexed for different functions. If the design does not strictly enforce access controls at the boundary of these shared resources, an untrusted agent may be able to access data or control signals meant exclusively for the trusted agent. This often happens when isolation logic relies on software configuration that can be altered by the untrusted agent or when the hardware default allows promiscuous access.

The root cause is typically a design flaw where the shared nature of the resource is not accompanied by a robust hardware-enforced access policy that considers the security context (TrustZone state, privilege level) of the requestor.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An untrusted agent can read sensitive data stored in shared memory or passing through a shared interconnect. |
| **Integrity** | An untrusted agent can modify data or configuration settings of a shared resource, potentially affecting the trusted agent's operation. |
| **Availability** | An untrusted agent can monopolize a shared resource or configure it in a way that denies service to the trusted agent (e.g., locking a shared bus). |

## Attack Scenario

An attacker controls the untrusted application processor on an SoC. The SoC design includes a Hardware Root of Trust (HRoT) that uses a specific block of local SRAM for its private operations and secure storage. However, this SRAM is also mapped into the global address space accessible by the application processor to facilitate inter-processor communication (mailbox functionality). The isolation logic intended to restrict the application processor's access to only the "mailbox" portion of the SRAM is flawed or misconfigured. In the next step, the attacker running code on the application processor issues read and write transactions to the memory addresses corresponding to the HRoT's private stack and key storage within that SRAM block. Because the isolation is insufficient, the hardware allows these transactions, enabling the attacker to extract the HRoT's secrets and compromise the device's root of trust.

## MITRE Reference Data

### Description

The System-On-a-Chip (SoC) does not properly isolate shared resources between trusted and untrusted agents.

### Extended Description

A System-On-a-Chip (SoC) has a lot of functionality, but it may have a limited number of pins or pads. A pin can only perform one function at a time. However, it can be configured to perform multiple different functions. This technique is called pin multiplexing. Similarly, several resources on the chip may be shared to multiplex and support different features or functions. When such resources are shared between trusted and untrusted agents, untrusted agents may be able to access the assets intended to be accessed only by the trusted agents.

### Demonstrative Examples

#### Example 1

Consider the following SoC design. The Hardware Root of Trust (HRoT) local SRAM is memory mapped in the core{0-N} address space. The HRoT allows or disallows access to private memory ranges, thus allowing the sram to function as a mailbox for communication between untrusted and trusted HRoT partitions.

We assume that the threat is from malicious software in the untrusted domain. We assume this software has access to the core{0-N} memory map and can be running at any privilege level on the untrusted cores. The capability of this threat in this example is communication to and from the mailbox region of SRAM modulated by the hrot_iface. To address this threat, information must not enter or exit the shared region of SRAM through hrot_iface when in secure or privileged mode.

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | Separation of Privilege | When sharing resources, avoid mixing agents of varying trust levels. Untrusted agents should not share resources with trusted agents. | - |

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement hardware-enforced isolation (e.g., TrustZone, PMP) at every shared resource boundary to prevent cross-trust-level access without explicit gating. | Critical |
| Architecture and Design | Assign each shared resource (SRAM, bus, peripheral) a dedicated access control filter that validates the security state of the requestor before granting access. | Critical |
| Implementation | Configure default-deny policies on all shared interconnects so that resources are inaccessible unless explicitly granted by trusted firmware during boot. | High |
| Testing | Perform penetration testing across trust boundaries on shared resources in both pre-silicon (emulation/FPGA) and post-silicon environments. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
