## CWE Explanation

This weakness occurs when the memory-mapped I/O (MMIO) register interface of a hardware component lacks appropriate access control checks, allowing unauthorized software to read sensitive data or modify critical configuration settings.

In System-on-Chip (SoC) designs, hardware peripherals (like crypto engines, interrupt controllers, or timers) expose their functionality to software via registers mapped into the system address space. Secure design principles dictate that sensitive registers—such as those holding encryption keys, root secrets, or privilege-level configurations—must only be accessible to trusted software running at a high privilege level (e.g., Machine Mode, EL3, or Secure World).

If the hardware logic implementing the register interface fails to validate the privilege level or security attribute of the incoming bus transaction, unprivileged or untrusted software can access these registers. This bypasses software-based security models and can lead to complete system compromise.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An attacker can read sensitive assets (keys, secrets) directly from hardware registers. |
| **Integrity** | An attacker can modify control registers to change the device state, disable security features, or escalate privileges. |
| **Availability** | An attacker can write invalid values to configuration registers, causing the hardware to hang, reset, or behave unpredictably. |

## Attack Scenarios

### Scenario 1: Cryptographic Key Leakage
A hardware AES accelerator is designed to be used by the kernel driver. It has a set of registers for loading the 128-bit encryption key. Ideally, these registers should be "Write-Only" or protected so that once a key is written, it cannot be read back by software. However, the hardware implementation incorrectly allows read access to these key registers from any privilege level. An attacker running a user-space application simply maps the physical address of the AES accelerator and issues a read instruction to the key register offsets, successfully extracting the secret key currently in use by the kernel.

### Scenario 2: Privilege Escalation via CSR
A RISC-V processor implementation includes a Machine Exception Program Counter (`mepc`) register, which stores the return address after handling a trap. This register is architecturally defined as accessible only in Machine Mode (M-Mode). However, the Control Status Register (CSR) decode logic has a bug: it fails to check the current privilege level when the target register is `mepc`. An attacker running unprivileged code (User Mode) writes a target kernel address into the `mepc` register using a standard CSR write instruction. The attacker then triggers a synchronous exception (e.g., an `ecall`). When the trap handler executes `mret` (return from trap), the processor jumps to the address the attacker planted in `mepc`, but now executing with the restored privileges (potentially M-Mode), effectively granting the attacker full control over the system.

## MITRE Reference Data

### Description

The product uses memory-mapped I/O registers that act as an interface to hardware functionality from software, but there is improper access control to those registers.

### Extended Description

Software commonly accesses peripherals in a System-on-Chip (SoC) or other device through a memory-mapped register interface. Malicious software could tamper with any security-critical hardware data that is accessible directly or indirectly through the register interface, which could lead to a loss of confidentiality and integrity.

### Demonstrative Examples

#### Example 1

The register interface provides software access to hardware functionality. This functionality is an attack surface. This attack surface may be used to run untrusted code on the system through the register interface. As an example, cryptographic accelerators require a mechanism for software to select modes of operation and to provide plaintext or ciphertext data to be encrypted or decrypted as well as other functions. This functionality is commonly provided through registers.

```other
(bad code)
Cryptographic key material stored in registers inside the cryptographic accelerator can be accessed by software.
```

```other
(good code)
Key material stored in registers should never be accessible to software. Even if software can provide a key, all read-back paths to software should be disabled.
```

#### Example 2

The example code is taken from the Control/Status Register (CSR) module inside the processor core of the HACK@DAC'19 buggy CVA6 SoC [REF-1340]. In RISC-V ISA [REF-1341], the CSR file contains different sets of registers with different privilege levels, e.g., user mode (U), supervisor mode (S), hypervisor mode (H), machine mode (M), and debug mode (D), with different read-write policies, read-only (RO) and read-write (RW). For example, machine mode, which is the highest privilege mode in a RISC-V system, registers should not be accessible in user, supervisor, or hypervisor modes.

```verilog
(bad code)
if (csr_we || csr_read) begin
	if ((riscv::priv_lvl_t'(priv_lvl_o & csr_addr.csr_decode.priv_lvl) != csr_addr.csr_decode.priv_lvl) && !(csr_addr.address==riscv::CSR_MEPC)) begin
	csr_exception_o.cause = riscv::ILLEGAL_INSTR;
	csr_exception_o.valid = 1'b1;
	end
	// check access to debug mode only CSRs
	if (csr_addr_i[11:4] == 8'h7b && !debug_mode_q) begin
	csr_exception_o.cause = riscv::ILLEGAL_INSTR;
	csr_exception_o.valid = 1'b1;
	end
end
```

The vulnerable example code allows the machine exception program counter (MEPC) register to be accessed from a user mode program by excluding the MEPC from the access control check. MEPC as per the RISC-V specification can be only written or read by machine mode code. Thus, the attacker in the user mode can run code in machine mode privilege (privilege escalation).

To mitigate the issue, fix the privilege check so that it throws an Illegal Instruction Exception for user mode accesses to the MEPC register. [REF-1345]

```verilog
(good code)
if (csr_we || csr_read) begin
	if ((riscv::priv_lvl_t'(priv_lvl_o & csr_addr.csr_decode.priv_lvl) != csr_addr.csr_decode.priv_lvl)) begin
	csr_exception_o.cause = riscv::ILLEGAL_INSTR;
	csr_exception_o.valid = 1'b1;
	end
	// check access to debug mode only CSRs
	if (csr_addr_i[11:4] == 8'h7b && !debug_mode_q) begin
	csr_exception_o.cause = riscv::ILLEGAL_INSTR;
	csr_exception_o.valid = 1'b1;
	end
end
```

## Remediations from MITRE

| Phase | Strategy | Mitigation | Effectiveness |
|-------|----------|------------|---------------|
| Architecture and Design | - | Design proper policies for hardware register access from software. | - |
| Implementation | - | Ensure that access control policies for register access are implemented in accordance with the specified design. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Implement privilege-level checks in the register decode logic that validate the current execution mode (e.g., Machine/Supervisor/User) against the required privilege for each register address. | Critical |
| Architecture and Design | Enforce write-only or no-read-back policies for registers containing cryptographic key material to prevent key extraction via register reads. | Critical |
| Implementation | Ensure all CSR and MMIO access control checks are performed without exceptions or carve-outs; remove any hardcoded bypasses for specific register addresses. | High |
| Testing | Develop comprehensive register access test suites that attempt reads and writes to every register from every privilege level and verify correct access denial. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
