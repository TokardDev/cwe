# CWE-1267 - Policy Uses Obsolete Encoding

## CWE Explanation

This weakness occurs when a system's access control policy relies on an encoding scheme or logic that has become obsolete, incorrect, or insufficient due to system evolution.

In hardware designs, security policies often use compact encodings to represent trust levels or access rights. For example, a single bit in a transaction identifier might define "Secure" vs. "Non-Secure", or a specific pattern of bits might map to a specific bus master. Over time, as the system architecture changes—new masters are added, trust models are updated, or address maps are expanded—the original encoding logic may no longer hold true.

If the hardware logic implementing the policy check is not updated to reflect these changes (e.g., it continues to use a legacy "shortcut" logic), it may inadvertently grant trusted privileges to a newly added untrusted component. This is common when design reuse (IP integration) brings legacy access control modules into a new SoC context without re-validating the underlying assumptions of the policy encoding.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An untrusted agent may be granted read access to secure assets because its ID accidentally matches a legacy trusted pattern. |
| **Integrity** | An untrusted agent may be allowed to modify critical settings if the policy logic incorrectly classifies it as trusted. |
| **Availability** | Denial of service is possible if an untrusted agent can access and disable shared resources due to the policy flaw. |

## Attack Scenario

A legacy SoC design used a simple rule for its bus fabric: "Any master with an odd ID is Trusted, and any master with an even ID is Untrusted." At the time, Master 1 (Trusted) and Master 2 (Untrusted) were the only agents.
In the next generation of the chip, designers add a new, untrusted network offload engine and assign it ID 3. They reuse the existing bus firewall logic without updating the policy encoding.
The firewall continues to apply the legacy rule: "ID is odd -> Trusted."
An attacker who compromises the firmware on the new network offload engine (Master ID 3) initiates a transaction to a secure memory region. The firewall checks the ID (3 is odd), incorrectly classifies the request as Trusted, and allows the access. The attacker thus gains unauthorized access to secure memory.

## MITRE Reference Data

### Description

The product uses an obsolete encoding mechanism to implement access controls.

### Extended Description

Within a System-On-a-Chip (SoC), various circuits and hardware engines generate transactions for the purpose of accessing (read/write) assets or performing various actions (e.g., reset, fetch, compute, etc.). Among various types of message information, a typical transaction is comprised of source identity (identifying the originator of the transaction) and a destination identity (routing the transaction to the respective entity). Sometimes the transactions are qualified with a Security Token. This Security Token helps the destination agent decide on the set of allowed actions (e.g., access to an asset for reads and writes). A policy encoder is used to map the bus transactions to Security Tokens that in turn are used as access-controls/protection mechanisms. A common weakness involves using an encoding which is no longer trusted, i.e., an obsolete encoding.

### Demonstrative Examples

#### Example 1

For example, consider a system that has four bus masters. The table below provides bus masters, their Security Tokens, and trust assumptions.

| Bus Master | Security Token Decoding | Trust Assumptions |
| :--- | :--- | :--- |
| Master_0 | "00" | Untrusted |
| Master_1 | "01" | Trusted |
| Master_2 | "10" | Untrusted |
| Master_3 | "11" | Untrusted |

The policy encoding is to be defined such that Security Token will be used in implemented access-controls. The bits in the bus transaction that contain Security-Token information are Bus_transaction [15:11]. The assets are the AES-Key registers for encryption or decryption. The key of 128 bits is implemented as a set of four, 32-bit registers.

Below is an example of a policy encoding scheme inherited from a previous project where all "ODD" numbered Security Tokens are trusted.

```other
(bad code)
If (Bus_transaction[14] == "1")
	Trusted = "1"
Else
	Trusted = "0"

If (trusted)
	Allow access to AES-Key registers
Else
	Deny access to AES-Key registers
```

The inherited policy encoding is obsolete and does not work for the new system where an untrusted bus master with an odd Security Token exists in the system, i.e., Master_3 whose Security Token is "11". Based on the old policy, the untrusted bus master (Master_3) has access to the AES-Key registers.
