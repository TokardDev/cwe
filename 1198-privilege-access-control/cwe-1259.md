# CWE-1259 - Improper Restriction of Security Token Assignment

## CWE Explanation

This weakness occurs in System-on-Chip (SoC) architectures where "Security Tokens" (also known as Master IDs, Source IDs, or Stream IDs) are used to identify the initiators of transactions but are not adequately protected against modification.

In many bus protocols (like AXI, AHB, or OCP), every read or write transaction carries a sideband signal denoting the identity or privilege level of the master that initiated it. Downstream components, such as memory protection units or peripheral firewalls, rely on this token to enforce access control policies. If the mechanism for assigning these tokens is mutable by the agent itself, or if an untrusted agent can spoof the token of a trusted agent, the entire security model collapses.

The root cause is typically a design flaw where the register responsible for holding the agent's Security Token is writable by that agent, or where the interconnect logic allows an agent to arbitrarily drive the token signals without validation.

## Impact Assessment

| Aspect | Impact |
|--------|--------|
| **Confidentiality** | An attacker can spoof a trusted token to bypass read protections and access sensitive secrets (e.g., encryption keys). |
| **Integrity** | An attacker can masquerade as a trusted writer to modify critical system configuration or secure memory. |
| **Availability** | An attacker can use a spoofed token to shut down critical components or reconfigure system resources, causing a denial of service. |

## Attack Scenario

An SoC design includes a Main Controller (trusted, Token ID `0x1`) and an Auxiliary Controller (untrusted, Token ID `0x2`). Access to the AES encryption engine's key registers is restricted by a hardware firewall that only permits transactions with Token ID `0x1`. However, the Auxiliary Controller's hardware interface includes a memory-mapped configuration register that determines the Token ID value driven onto the bus for its transactions. This register is not locked and is writable by the firmware running on the Auxiliary Controller. In the next step, an attacker who has compromised the Auxiliary Controller writes `0x1` to this configuration register. Subsequent transactions initiated by the Auxiliary Controller now carry the trusted Token ID `0x1`, allowing the attacker to bypass the firewall and read the AES keys.

## MITRE Reference Data

### Description

The System-On-A-Chip (SoC) implements a Security Token mechanism to differentiate what actions are allowed or disallowed when a transaction originates from an entity. However, the Security Tokens are improperly protected.

### Extended Description

Systems-On-A-Chip (Integrated circuits and hardware engines) implement Security Tokens to differentiate and identify which actions originated from which agent. These actions may be one of the directives: 'read', 'write', 'program', 'reset', 'fetch', 'compute', etc. Security Tokens are assigned to every agent in the System that is capable of generating an action or receiving an action from another agent. Multiple Security Tokens may be assigned to an agent and may be unique based on the agent's trust level or allowed privileges. Since the Security Tokens are integral for the maintenance of security in an SoC, they need to be protected properly. A common weakness afflicting Security Tokens is improperly restricting the assignment to trusted components.

### Demonstrative Examples

#### Example 1

For example, consider a system with a register for storing an AES key for encryption and decryption. The key is of 128 bits implemented as a set of four 32-bit registers. The key register assets have an associated control register, AES_KEY_ACCESS_POLICY, which provides the necessary access controls. This access-policy register defines which agents may engage in a transaction, and the type of transaction, with the AES-key registers. Each bit in this 32-bit register defines a security Token. There could be a maximum of 32 security Tokens that are allowed access to the AES-key registers. The number of the bit when set (i.e., "1") allows respective action from an agent whose identity matches the number of the bit and, if "0" (i.e., Clear), disallows the respective action to that corresponding agent.

Let's assume the system has two agents: a Main-controller and an Aux-controller. The respective Security Tokens are "1" and "2".

```other
(bad code)
The Aux-controller could program its Security Token to "1" from "2".
```

The SoC does not properly protect the Security Token of the agents, and, hence, the Aux-controller in the above example can spoof the transaction (i.e., send the transaction as if it is coming from the Main-controller to access the AES-Key registers)

```other
(good code)
The SoC needs to protect the Security Tokens. None of the agents in the SoC should have the ability to change the Security Token.
```

| Architecture and Design; Implementation | - | Security Token assignment review checks for design inconsistency and common weaknesses. Security-Token definition and programming flow is tested in both pre-silicon and post-silicon testing. | - |

*Source: MITRE CWE Database*

## Recommended Remediations

| Phase | Remediation | Priority |
|-------|-------------|----------|
| Architecture and Design | Ensure all Security Token values are hardwired in hardware (e.g., via tie-offs or OTP fuses) and are not writable by any software or firmware running on the agent. | Critical |
| Architecture and Design | Implement token validation logic at the bus fabric level that cross-checks the physical port of origin against the declared Security Token to detect spoofing. | Critical |
| Implementation | Lock all token configuration registers during the secure boot phase using write-once bits that cannot be cleared until the next power-on reset. | High |
| Testing | Create dedicated pre-silicon and post-silicon test cases that attempt to modify Security Tokens from each agent and verify that all such attempts are blocked. | High |

*These remediations are recommended best practices and are not sourced from MITRE.*
